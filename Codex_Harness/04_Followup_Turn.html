A follow-up turn — why turn 2 barely re-injects anything

The second message in a session runs almost the same machinery as the first — with one big exception: the context-injection step goes quiet. Here's exactly what changes, and the subtle reason the model still remembers everything.

The one-line difference

Turn 1 sets the context. Every follow-up turn skips re-injecting it and appends only diffs — usually nothing at all.

The model still sees your memories, AGENTS.md, and environment on turn 2 — not because they're re-sent, but because they're already in the conversation history, which is replayed in full on every single request (clone_history().for_prompt()). Re-injecting them would be pure waste.

Continuing the example

Same session as the walkthrough. You already asked this, and Codex did it:

turn 1 — "Fix the failing test in tests/parse_test.rs." ✓ done

Now you type your follow-up:

"Now add a docstring to that function and run the test again to be sure."
✔ Confirmed by a real captured trace. Resuming a real session, turn 2 took 4 model round-trips vs turn 1's 11, with no re-exploration. See the Real Trace.

02 What's different · the diff-only context path

On turn 1, record_context_updates_and_set_reference_context_item (mod.rs:3549) saw reference_context_item == None, built the whole bundle, and stashed a baseline at the end (mod.rs:3631). On turn 2 that baseline exists, so it takes the steady-state branch and asks three narrow questions:

1

Did any setting change?

build_settings_update_items (updates.rs:232) emits a small developer message only if one of these differs from the baseline: model switched, approval / sandbox / exec-policy changed, collaboration mode, multi-agent mode, realtime, or personality.

our example: nothing changed → emits nothing
2

Did the environment change?

update_world_state (history.rs:88) → render_history_diff(baseline, …) emits a fragment only for what actually moved — you cd'd, the git branch flipped, the working tree changed.

our example: maybe a 1-line "git status changed" note, or nothing
3

Did the turn-context itself change?

If so, build_turn_context_contribution_items runs. Otherwise skipped. Then the short-circuit (mod.rs:3606): if nothing changed and there's no new world-state item, it returns immediately, recording nothing.

our example: short-circuits
Net result for turn 2

The only new thing appended to history before the model is called is your message. No memories re-injected. No AGENTS.md re-injected. No environment block re-injected — they're all still in history from turn 1.

03 The contrast, at a glance

TURN 1  (reference_context_item = None → full inject)
  history before model:
    [ permissions + AGENTS.md + MEMORY summary + skills + environment ]   ← the big bundle
    [ your message #1 ]

TURN 2  (reference_context_item = Some → diff only)
  history before model:
    …everything from turn 1 is still here (bundle + msg#1 + all tool calls + answer)…
    [ settings diff?   → empty, nothing changed ]
    [ world-state diff? → empty (or a 1-line git-status note) ]
    [ your message #2 ]                                             ← usually the ONLY new item
Verified against a real run

Resuming the same session and sending a real turn 2, the internal trace showed resume_thread_with_history (the reconstruction path) and zero full-context re-injection — the only full_context hits were the full_context_window_limit token field, not a context bundle. The follow-up made 4 model round-trips against turn 1's 11, and the memory-consolidation agent did not spawn this time (it runs periodically, not every turn).

04 Two quieter differences worth knowing

previous_turn_settings is now populated

Turn 1 set it at its end (set_previous_turn_settings). This is what enables the model-switch / context-downshift detection in run_pre_sampling_compact — so it's on follow-up turns, not the first, that switching models can trigger a pre-turn compaction against the old model.

Compaction becomes the real story over time

History only grows. The token check does nothing on turn 1, occasionally fires on turn 5, and reliably fires on turn 30 — summarizing old history into a Compaction item before the request. The first turn you'll never see it; follow-up turns are where compaction lives.

05 Turn 2's loop, for completeness

Once setup is done, the loop is ordinary — the same machinery as turn 1.

step 1: stream → apply_patch (add docstring to src/parse.rs)
          → assess_patch_safety → writable path + sandbox → auto-approve → edit  → end_turn=false ↻
step 2: stream → shell `cargo test parse` → execpolicy → sandbox → passes      → end_turn=false ↻
step 3: stream → "Done — added the docstring and the test still passes."       → end_turn=TRUE ■
Why "that function" just works

The model had full continuity into turn 2 — it knew which function "that function" referred to — precisely because turn 1's entire trace (context bundle, tool calls, the fix) was replayed in the prompt.

06 When a follow-up turn does re-inject

The diff path isn't always empty. A small developer "context update" message is appended at the top of a follow-up turn when you change something — each is a targeted one-message diff, never the whole bundle again.

switch modelsmodel-instructions update (+ maybe a pre-turn compaction)
change approval / sandboxa permissions update (via /approvals etc.)
cd, or git branch / worktree changesa world-state diff
toggle personality / collaboration modethe matching mode update
