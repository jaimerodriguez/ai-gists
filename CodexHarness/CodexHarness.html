The life of a turn

One trip around the loop, in plain language — from the moment you hit Enter to the moment Codex answers. Everything after this section is the same story told in depth.

1
A message arrivesOp::UserInput

You type and hit Enter. The frontend drops your message onto the submission queue; the session picks it up and starts a turn. From here on, everything the model, the tools, and the UI say travels back out on the event queue.

2
The system prompt is chosenbase_instructions

Not rebuilt from scratch — chosen by priority: your config override, else the prompt this thread was started with, else the model's own built-in instructions (with the personality block spliced in). This ~21 KB of text becomes the request's top-level instructions and is deliberately kept out of the conversation so it stays a stable, cacheable header.

3
The context is assembled

Everything that isn't the system prompt becomes ordered messages in front of your request. In this exact order:

3.1Permissions & sandbox are written out — twice: prose for the model to read, and precise XML for the machine.
3.2The environment is captured: working directory, shell, date, timezone, and which folders are writable.
3.3Project instructions (AGENTS.md) are found by walking up to the repo root and stitched together, root → cwd.
3.4Skills & memories are appended — the skill catalog, plus anything a mentioned skill or a memory provider injects for this turn.
3.5Connected apps (MCP servers / connectors) are described, if they're enabled.
3.6Your actual message is placed on the end.
3.7A timestamp is slipped in right before sending, so the model always knows "now."
4
The toolbox is laid outbuilt_tools

The harness works out which tools this model and config are allowed to see — shell, file-patching, web search, plan updates, MCP tools — and writes them as JSON specs. In the traced run that was 17 tools. Big tool catalogs are hidden behind a searchable index so the prompt stays lean.

5
The request is sealed and sentResponsesApiRequest

All of the above is packed into one request, stamped with identity headers and a thread-stable cache key, and streamed to the backend — over a WebSocket when possible, falling back to plain HTTP if not.

6
The answer streams back

Tokens arrive live and the assistant's text appears as it's written. As each piece of the reply finalizes, the harness asks one question: is this a tool call, or just a message?

7
If it's just a message → the turn is done

A bare reply with no tool call ends the turn. Stop-hooks get a veto (they can force a continuation); if none object, the loop breaks and the answer is handed back to you.

8
If it's a tool call → it runs, carefully

Every tool the model asks for goes through the same gauntlet before anything touches your machine:

8.1The call is parsed into a typed request and recorded to history immediately.
8.2Pre-tool hooks get a chance to rewrite or block it.
8.3The command is classified: auto-approve, ask first, or outright forbidden.
8.4Permission is checked when needed — either an automated guardian LLM reviews it, or you're asked. The tool pauses, parked on a channel, until the decision comes back.
8.5The sandbox is applied (Seatbelt, landlock, or a restricted token) and the command runs inside it.
8.6If the sandbox blocks it, the harness may ask to retry with less confinement — never silently.
8.7Post-tool hooks run, the output is truncated to fit, and the result is written back to history.
9
The loop turns

The tool's output becomes the next request's input, and we go back to step 5 — the model sees what happened and picks its next move. Several tools in one reply run in parallel but report back in the order they were asked.

10
Context is watched the whole time

After each round the harness checks how full the context window is. If it's filling up, the thread is compacted — summarized down to a checkpoint — before the loop continues, so a long task never simply runs out of room.

11
The turn closes and is saved

When the model finally answers with no tool call, the turn ends, the reply is surfaced, and the entire exchange is appended to the session's rollout file on disk — the durable record that makes resume possible.

12
Next time, it resumes

Your follow-up message replays the whole transcript as input — but because the cache key is unchanged, almost all of it is served from cache (92–98% in the trace). You pay for the long history essentially once.

That's the spine. Steps 2–5 are the request build (§02); steps 6–9 are the loop and stream (§03–§04); step 8 is the safety gate (§06); step 10 is compaction (§07); steps 11–12 are persistence (§09). Read on for any one of them in full.

§ 01

Architecture: four frontends, one core, two protocol enums

Everything the harness does hangs off a single mental model: a Session that consumes a queue of Op submissions and emits a queue of EventMsg events. Every frontend and every sub-agent speaks that same two-queue protocol.

The workspace splits along clean seams. The networking layer was lifted out of core into two dedicated crates, leaving core to own orchestration:

LayerCrate / dirOwns
Raw transportcodex-clientHTTP/SSE primitives, reqwest builder, retry policy, proxy & CA handling
Typed APIcodex-apiResponsesApiRequest, SSE→ResponseEvent mapping, rate-limit parsing, Provider URLs
Orchestrationcodex-coreSession, turn loop, tool router, sandbox, compaction, auth & header wiring
Protocolcodex-protocolOp, EventMsg, ResponseItem, policy enums, base prompt files
Frontendstui · exec · app-server · mcpAttach to a Session's submission/event queues

The two queues

A Session owns two channels session/mod.rs:539. The submission queue (SQ) is bounded at 512 and carries Op commands inward; the event queue (EQ) is unbounded and carries EventMsg events outward. A submission_loop session/handlers.rs:703 matches on sub.op and dispatches. This is the entire external contract — and the reason delegation composes: a sub-agent is just a parent consuming a child's EQ and producing into its SQ (see §9).

# the whole harness, in five hops
Op::UserInput ──▶ submission_loop ──▶ Session::spawn_task(RegularTask)
                                          │
                                          ▼
                              RegularTask::run ──▶ run_turn ──loop──▶ run_sampling_request
                                          │                                 │
                              EventMsg ◀──┴── stream events ◀── SSE ◀───────┘

The path from a user message to a running task is short: user_input_or_turn resolves the active turn context and calls spawn_task(…, RegularTask::new()) session/handlers.rs:260. RegularTask::run emits TurnStarted inline, then calls run_turn in a loop that re-fires while pending input remains tasks/regular.rs:73.

§ 04

The streaming state machine

A single match inside try_run_sampling_request session/turn.rs:1986 consumes the SSE stream event by event, turning wire events into internal ResponseEvents and firing tool futures the instant a call completes.

Two mapping layers sit between the socket and the loop. First, process_responses_event codex-api/sse/responses.rs:325 maps wire event types to codex_api::ResponseEvent; response headers are drained into synthetic events (ServerModel, RateLimits, ModelsEtag) before the body loop begins. Second, map_response_events client.rs:1898 buffers OutputItemDone items so the next WebSocket request can send only a delta (§8).

Captured · SSE sequence, request #030 39 events, in this order:
response.created
response.in_progress
response.output_item.added        # assistant message begins
response.content_part.added
response.output_text.delta   ×27   # streamed tokens
response.output_text.done
response.content_part.done
response.output_item.done          # message finalized → recorded to history
response.output_item.added         # function_call begins
response.function_call_arguments.delta
response.function_call_arguments.done
response.output_item.done          # tool call finalized → tool future spawned
response.completed                  # token_usage, end_turn → loop decision

The critical move: dispatch on output_item.done

When an item finalizes, handle_output_item_done stream_events_utils.rs:405 runs ToolRouter::build_tool_call. If it's a tool call, the item is recorded to history immediately, and a boxed future is pushed onto an in-flight FuturesOrdered:

// stream_events_utils.rs:436
let tool_future = Box::pin(
    ctx.tool_runtime.clone().handle_tool_call(call, cancellation_token));
output.needs_follow_up = true;
output.tool_future = Some(tool_future);

Because the stream keeps flowing while tools run, a model can emit several calls in one response and they execute concurrently — but FuturesOrdered session/turn.rs:1926 yields results in emission order, so the function_call_output items appended to history preserve the order the model asked for. On response.completed, remaining in-flight tools are drained session/turn.rs:2364 before the loop returns.

The stream terminates on response.completed (returns immediately) or errors: a byte-stream ending early yields "stream closed before response.completed", and a stall past the idle timeout (default 300,000 ms) yields "idle timeout waiting for SSE" codex-api/sse/responses.rs:502.

§ 05

The tool system

Four concerns kept separate: the spec the model sees, the runtime that executes, the registry that maps names, and the router that schedules. A single dispatch funnel unifies them.

The 17 tools in the captured request are computed per-request by build_tool_specs_and_registry tools/spec_plan.rs:171, gated by model family and feature flags. Visibility is a first-class enum, not a boolean tools/tool_executor.rs:15:

ExposureMeaning
DirectIn the initial model-visible list (and code-mode nested surface)
DeferredRegistered but hidden; discoverable via tool_search — keeps huge MCP catalogs out of the prompt
DirectModelOnlyModel-visible but excluded from code-mode
HiddenDispatchable, never advertised (e.g. legacy shell_command under unified_exec)

Two tool shapes in the capture

exec_command is the unified-exec shell tool — a function spec with a rich schema (cmd, workdir, yield_time_ms, sandbox_permissions, …). It returns a session id so a follow-up write_stdin can resume the same live PTY; up to 64 processes persist across calls unified_exec/mod.rs:1.

apply_patch is the instructive one — a custom (freeform) tool constrained by a Lark grammar sent inline in the spec:

// captured apply_patch.format
{ type: "grammar", syntax: "lark", definition:
  "start: begin_patch hunk+ end_patch
   add_hunk: \"*** Add File: \" filename LF add_line+
   update_hunk: \"*** Update File: \" filename LF change_move? change? …" }

Its payload arrives as a custom_tool_call with an input string, parsed by codex_apply_patch::parse_patch. The parser is deliberately lenient (PARSE_IN_STRICT_MODE = false) to tolerate model whitespace quirks apply-patch/parser.rs:53.

Routing, dispatch, parallelism

build_tool_call tools/router.rs:112 parses a finalized ResponseItem into a typed ToolPayload (Function, Custom, or ToolSearch). Every call then passes through one funnel — dispatch_any_with_terminal_outcome tools/registry.rs:405 — which centralizes telemetry, payload-kind validation, PreToolUse / PostToolUse hooks, and error shaping, so handlers stay tiny.

Parallelism is governed by a single shared RwLock tools/parallel.rs:133: parallel-safe tools take a read lock (concurrent); anything else takes a write lock (globally exclusive). A tool opts in via supports_parallel_tool_calls() — true for unified-exec and read-only MCP tools. That is exactly the RT-3 behavior seen in the trace: two exec_command probes ran concurrently.

§ 06

The safety & execution gate

Between "model asked to run a command" and "the command runs" sits a pipeline: classify → approve → sandbox → execute → escalate-on-failure. Its cleanest idea is keeping two policy axes orthogonal.

Two axes, one cross-product

Approval policy (when to ask a human) is independent of sandbox policy (what a process may touch). AskForApproval protocol.rs:901 is UnlessTrusted | OnRequest | Granular | Never; SandboxPolicy protocol.rs:988 is ReadOnly | WorkspaceWrite | ExternalSandbox | DangerFullAccess, lowered to a runtime PermissionProfile models.rs:408. The captured session ran workspace-write + never — auto-approve but always sandboxed (the metadata confirmed sandbox:"seatbelt").

Every command collapses to one closed enum via create_exec_approval_requirement_for_command exec_policy.rs:269: Skip | NeedsApproval | Forbidden. Commands are canonicalized first command_canonicalization.rs:14 so /bin/bash -lc and bash -lc share one approval key — but exact script text is preserved when it can't be safely tokenized.

Sandbox-first, escalate on failure

Commands run sandboxed by default (Seatbelt on macOS, bubblewrap+seccomp on Linux, restricted-token on Windows). Only when execution is detected as a sandbox denial sandboxing/denial.rs:6 does ToolOrchestrator::run tools/orchestrator.rs:134 consider a reduced-sandbox retry — and it refuses that retry when it would silently drop denied-read restrictions.

The guardian: an optional, fail-closed LLM reviewer

When approval policy is OnRequest/Granular and the reviewer is set to auto guardian/review.rs:168, a would-be human prompt is instead routed to a locked-down read-only, approval=never review session. It returns strict JSON { outcome: allow|deny, … }, treats the parent transcript as untrusted evidence, and fails closed to Deny on any timeout/parse error guardian/mod.rs:1. A circuit breaker interrupts the turn after repeated denials. Crucially, the guardian replaces the prompt — never the sandbox.

The suspend/resume primitive

An approval parks the tool future on a registered oneshot: insert tx into pending_approvals[id], emit ExecApprovalRequest, then rx.await.unwrap_or(Abort) session/mod.rs:2194. The frontend's Op::ExecApproval{id, decision} calls notify_approval, which sends on the channel and unparks the future. A dropped channel defaults to Abort — the safe default.

§ 07

Context-window management

Token accounting, auto-compaction, and truncation — the machinery that keeps a long thread inside the window without a client-side tokenizer.

Accounting: trust the server, estimate the delta

The "current context size" the harness reasons about is the server's last-turn total_tokens plus a coarse byte-based estimate of items appended locally since context_manager/history.rs:328 (≈4 bytes/token). Two ceilings gate compaction session/context_window.rs:65: an auto-compact limit at min(config, 90% of window) openai_models.rs:444, and a full-window backstop at ~95% effective.

Compaction: three implementations behind one dispatcher

run_auto_compact session/turn.rs:917 picks a strategy:

VariantMechanismSite
Token-budgetNo summary — just installs a fresh window & re-injects contextcompact_token_budget.rs
Remote v2Streams a normal request with a CompactionTrigger sentinel; expects exactly one Compaction output itemcompact_remote_v2.rs
Remote v1Dedicated /responses/compact endpoint returns new historycompact_remote.rs
LocalClient appends a summarization prompt, drains to completion, rebuilds historycompact.rs
Captured · beta header The session advertised x-codex-beta-features: remote_compaction_v2 on every request — the remote-v2 path was armed (though the short task never hit the limit to trigger it).

Compaction is modeled as a replayable checkpoint, not a mutation: history replacement is persisted as a Compacted rollout item carrying the full replacement_history plus monotonic window ids session/mod.rs:3007. Resume/fork reconstructs exact state by scanning newest-first to the last checkpoint and replaying only the suffix session/rollout_reconstruction.rs:113 — O(suffix), not O(history).

Truncation: middle-out, at ingest

Large tool outputs are truncated when recorded, not at prompt-build time context_manager/history.rs:370, using the model's TruncationPolicy scaled ×1.2 for serialization overhead. The truncator keeps head and tail, cutting the middle with a visible marker — so both the command echo and the final error/result survive utils/output-truncation/lib.rs:25.

§ 08

Transport & resilience

One wire protocol (Responses API), two transports (WebSocket preferred, HTTP fallback), two tiers of retry, and an incremental-request trick that leans on the persistent socket.

URL & auth routing

The base URL is chosen by auth mode model-provider-info/lib.rs:241: ChatGPT-family auth → https://chatgpt.com/backend-api/codex; API-key → https://api.openai.com/v1. The captured session used ChatGPT auth (hence chatgpt-account-id and the /backend-api/codex/responses path).

Captured · WebSocket→HTTP fallback, observed live The trace shows both GET /codex/responses returning 101 Switching Protocols (WebSocket upgrade, header openai-beta: responses_websockets=2026-02-06) and subsequent POST /codex/responses returning 200 with SSE bodies. Because the logging proxy could not tunnel WS binary frames, the session degraded to HTTP SSE — inadvertently exercising the exact fallback path in the source: on retry-budget exhaustion the transport switches WebSocket→HTTPS and resets the retry counter responses_retry.rs:31.

Incremental request reuse over WebSocket

When the socket persists, get_incremental_items client.rs:1130 checks that all non-input fields are unchanged and the new input is a strict extension of previous_request.input ++ last_response.items_added; if so it sends only the delta with previous_response_id. The equality check is exhaustive field-by-field client.rs:300 so any schema change forces a full resend rather than a silent stale reuse.

Two-tier retry

Transport-level retries (429/5xx/network, jittered exponential backoff) sit under a turn-level retry that, on exhaustion, downgrades the transport and resets its budget. The SSE layer classifies errors explicitly codex-api/sse/responses.rs:374 — context_length_exceeded, insufficient_quota, cyber_policy abort; everything else is retryable with a server-parsed retry-after delay.

Captured · cache economics Across turn 1 the input grew 3→18 items, yet the final request cached 13,696 / 14,037 input tokens (97.6%). The stable prompt_cache_key = thread_id plus prefix-stable ordering is what buys this — the 21 KB base prompt is paid for essentially once per thread.
§ 09

Persistence & sub-agents

Every session is an append-only JSONL rollout; SQLite is a rebuildable index over it. Sub-agents are full sessions behind the same two-queue interface.

Rollout: JSONL is the source of truth

Sessions write to ~/.codex/sessions/YYYY/MM/DD/rollout-<ts>-<uuid>.jsonl rollout/recorder.rs:1498, one RolloutLine per line, flushed after each write. A persist-policy allow-list rollout/policy.rs:6 decouples the firehose sent to frontends from the curated subset written to disk — noisy streaming/begin events are dropped; messages, reasoning, patch results, token counts, and lifecycle survive. SQLite is filesystem-first-reconciled and always falls back to a file scan rollout/recorder.rs:401.

Captured · resume worked off the rollout Turn 2 was launched as a separate process via codex exec resume --last. It reconstructed the full turn-1 history from the JSONL rollout and replayed it as the 20-item input — the persistence layer and the resume path, verified end-to-end.

Delegation as a composed session

A sub-agent (review, compaction, guardian, collaboration) is spawned via run_codex_thread_interactive codex_delegate.rs:76, inheriting the parent's services but with SessionSource::SubAgent. The parent forwards the child's events and pipes its own Ops inward — except approval requests, which are routed to the parent's policy codex_delegate.rs:277. Review mode tasks/review.rs:96 is the canonical example: it disables web-search and collaboration, forces approval=never and a review rubric, runs one-shot, and folds the parsed ReviewOutputEvent back into the parent transcript.

§ 10

Transferable patterns

The design decisions worth stealing for any agent harness, distilled from the sections above.

P-01

Keep instructions out of the transcript

The base prompt is a separate field mapped to the API instructions slot, not concatenated into history — so the cacheable system prompt stays out of the mutable input and a thread-stable cache key can do its job.

P-02

Prefix-stable ordering, diff-based steady state

Inject full context once, then emit only diffs, with fixed leading order and a stable cache key. The payload confirms the payoff: 97.6% cache hits.

P-03

Concurrency with deterministic order

FuturesOrdered + a shared read/write lock gives parallel tool execution while outputs are appended in model-emission order.

P-04

One dispatch funnel

Route every tool call through a single function that owns telemetry, hooks, payload validation, and error shaping. Handlers stay trivial and uniform.

P-05

Two orthogonal policy axes

Separate "when to ask a human" from "what a process may touch." The same command can be auto-approved-but-sandboxed, prompted, or forbidden by the cross-product.

P-06

Sandbox-first, escalate on detected failure

Run sandboxed by default; only a detected denial triggers a reduced-sandbox retry — and never one that drops read restrictions.

P-07

LLM reviewer as a fail-closed gate

An optional reviewer replaces the human prompt, runs isolated and read-only, treats the transcript as untrusted, and defaults to Deny on any error.

P-08

Suspend on a oneshot, resume by id

Park the tool future on a registered channel, emit an event, await; external notify(id, decision) unparks it. Dropped channel = Abort.

P-09

Compaction as a replayable checkpoint

Persist the replacement history + window ids so resume reconstructs state in O(suffix), not O(history), by scanning to the last checkpoint.

P-10

JSONL truth + rebuildable index

Append-only rollout is canonical; SQLite is a reconciled accelerator that always falls back to a file scan. A persist allow-list keeps rollouts compact.

P-11

Exposure as an enum, not a bool

Direct/Deferred/DirectModelOnly/Hidden lets huge MCP catalogs stay reachable via search without bloating the initial tool list.

P-12

Sub-agents are just sessions

Delegation = a parent consuming a child's event queue and producing into its submission queue, intercepting only approvals. Review/compaction reuse one primitive.

§ 12

How this was captured

The live evidence is not a mock. It is the real codex binary at commit be33f80b talking to the production backend, teed through a logging proxy.

  1. Built the debug codex binary from this checkout and pointed it at an isolated CODEX_HOME carrying a real ChatGPT auth token.
  2. Set chatgpt_base_url and openai_base_url to a local Python reverse proxy that forwards to chatgpt.com and tees every request body and SSE response to disk (auth headers redacted, never written).
  3. Disabled request compression (features.enable_request_compression=false) so bodies were readable JSON rather than zstd.
  4. Ran a two-turn session in a throwaway git workspace: turn 1 created and tested a fib.py; turn 2 (resume --last) extended it — exercising the loop, tool dispatch, and cross-turn resume.
  5. Dissected the captured payloads (9 sampling requests, ~40 SSE events each) and cross-referenced every field to the source with five parallel source-reading passes.

The one honest caveat: because a plain HTTP proxy can't tunnel WebSocket frames, sampling traffic ran over the HTTP SSE fallback rather than the preferred WebSocket transport — which, as noted in §8, is itself a faithful exercise of the fallback code path.
