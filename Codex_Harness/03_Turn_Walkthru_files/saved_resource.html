How a Codex turn actually runs — one message, traced call by call

The big idea up front: a "turn" is a loop, not a single model call. You send one message; the model drives itself forward — read, run, edit, re-run, answer — and Codex keeps looping until the model says it's done. Here's that loop, followed through one concrete example.

⚠ Illustrative example. We later ran a real prompt and captured the trace — see the Real Trace. Where reality differed, it's flagged below.
The example we'll trace

A fresh session, in a git repo with an AGENTS.md and a couple of saved memories. You type:

"Fix the failing test in tests/parse_test.rs."
cwd ~/myproject approval OnRequest (default) sandbox workspace-write has AGENTS.md has memories

Act 1 From your keystroke to the turn engine

Before run_turn is even reached, your text takes a short trip through the queue.

1

Your text becomes a command

The TUI wraps your message as an Op::UserInput and calls CodexThread::submit(op). An Op is "a command for the core." Submitting just drops it on a queue and returns an ID — it doesn't block.

2

The queue turns it into a task

A submission loop (session/handlers.rs) picks it off the queue, sees it's user input, and calls Session::spawn_task with a RegularTask. (Compaction and review are other task types.)

3

The task calls run_turn

RegularTask::run (tasks/regular.rs:37) emits TurnStarted to the UI, then calls run_turn. It only re-calls it if you queued more input while it worked; normally it runs once.

Plain english

Your message becomes a queued command, the core turns that command into a "task," and the task calls run_turn. One user message = one run_turn.

Act 2 run_turn sets the table

Everything below happens once, before the model is ever called (turn.rs:142).

1

Get a model connection

model_client.new_session() — a turn-scoped client that caches the WebSocket / routing so retries within this turn reuse it.

2

Compact if we're already too full

run_pre_sampling_compact (:797). On a fresh session this does nothing — but it's where Codex would summarize old history if you were near the context limit, or if you'd just switched models. This step can itself call the model.

3

Inject the base context ← the big one

record_context_updates_and_set_reference_context_item (mod.rs:3549) → on a fresh session runs build_initial_context…, assembling one big bundle:

  • the permissions block (tells the model it's in workspace-write / OnRequest),
  • your AGENTS.md instructions,
  • the skills catalog and plugin list,
  • your memories — the summary file, read off disk and injected here,
  • the environment — cwd, sandbox mode, workspace roots, git status.
About that memory step

Only the memory summary is injected, and only once for the whole session — not the full memory files, and not every step. The model pulls deeper memory files later with memory tools if it needs them. Later turns append only changes to this bundle.

4

Inject anything your message specifically triggered

build_skills_and_plugins (:510) scans your text for explicit @skill or plugin:// mentions and injects only those. Your message has none, so nothing extra.

5

Run input hooks and record your message

run_hooks_and_record_inputs (:481). Any hooks you configured can inspect or block your input; then your message is recorded as pending input.

History now holds: [ big context bundle ] → [ your user message ]. Enter the loop.

Act 3 The loop — one step at a time

Each loop iteration = one request to the model.

In the real trace, this was different

The 5 clean steps below are illustrative. When we ran an actual prompt and captured it (Real Trace), the same machinery held but the behavior was richer: the loop made 11 model round-trips, not 5; it opened with a recon phase (pwd, find, git status, rustc --version) before writing anything; the shell tool was actually exec_command (the unified_exec runtime), not shell; the model adapted to the sandbox (it went dependency-free rather than fetch a crate); and a second thread ran in parallel doing memory-maintenance work in ~/.codex/memories/. The control flow was exactly as described — the step count and specific commands are illustrative.

One iteration of run_turn's loop (:225) does four things: drain pending input → add reminders → build the model input (for_prompt — mechanical cleanup only, no paraphrasing) → call run_sampling_request, which assembles the tools, builds the prompt, opens the stream, and reacts to what the model emits. Watch the end_turn flag — that's what decides whether we loop again.

step 1

The model reads the test file

Stream opens. The model emits a shell tool call: cat tests/parse_test.rs. Codex spawns it immediately as a background task and keeps reading the stream. When the response completes, drain_in_flight waits for the read and records the file contents into history.

end_turn = false → loop again ↻
step 2

The model runs the test

Same machinery, now with the file in context. The model emits shell → cargo test parse. This is where security fires (see Act 4). The test runs, fails, output goes back into history.

end_turn = false → loop again ↻
step 3

The model edits the code

The model emits an apply_patch call to fix src/parse.rs. apply_patch is safety-checked its own way (Act 4). The patch applies.

end_turn = false → loop again ↻
step 4

The model re-runs the test

Another shell → cargo test parse. Passes this time.

end_turn = false → loop again ↻
step 5

The model answers

A final assistant message — "Fixed it — the parser wasn't handling empty input; I added a guard." No tool call.

end_turn = TRUE → stop, return the answer ■
Five model requests, one user turn · 11 in the real run

If the model had emitted two independent reads in one response, they'd run concurrently — parallel-safe tools take a shared read-lock; a mutating tool like apply_patch takes an exclusive write-lock. All results are collected after the stream finishes, then fed back on the next loop.

Act 4 The instant a tool call fires — the security gate

Every tool call, the moment it's spawned, goes through the ToolOrchestrator (orchestrator.rs:134) — "approval → sandbox → run → maybe retry." The two tools in our trace check safety differently.

cargo test parse — a shell command
  1. Needs approval? Checked against the execpolicy rules → Allow / Prompt / Forbidden. Running tests is allowed under OnRequest → no prompt.
  2. Pick a sandbox. workspace-write + this OS → macOS Seatbelt (Landlock on Linux). Confined to the workspace, network off.
  3. Run it inside that sandbox.
  4. If denied (e.g. it tried the network): under OnRequest, Codex asks you; if you approve, it retries with the sandbox relaxed. Under Never it wouldn't ask — it hands the error back to the model.
editing src/parse.rs — apply_patch
  1. apply_patch skips execpolicy and calls assess_patch_safety (safety.rs:32).
  2. Every edited path inside a writable root? src/parse.rs is in the workspace → yes.
  3. Can we enforce a sandbox here? Yes → AutoApprove inside a sandbox, no prompt. (Sandbox guards against tricks like symlinks pointing outside the workspace.)
  4. If the file were outside the workspace (say ~/.ssh/config) → AskUser, or Reject outright under a Never policy.
The one-line rule

Nothing is security-checked when the prompt is built. Everything is checked at the moment each tool call executes — tool by tool.

your message
  → submit(Op::UserInput) → queue → RegularTask::run → run_turn

      SET THE TABLE (once):
        new model session
        pre-turn compaction?  (no, fresh session)
        inject base context:  permissions + AGENTS.md + MEMORY summary + skills + environment
        inject @mentions from your text  (none)

      THE LOOP (once per model request):
        step 1: prompt → stream → shell `cat test` → run (sandbox, read-only) → end_turn=false ↻
        step 2: stream → shell `cargo test` → approval? → sandbox → fails    → end_turn=false ↻
        step 3: stream → apply_patch → assess_patch_safety → auto-approve → edit → end_turn=false ↻
        step 4: stream → shell `cargo test` → passes                     → end_turn=false ↻
        step 5: stream → final answer                                    → end_turn=TRUE ■

  → TurnComplete

Burn in two things

A turn is a loop, not a call

The model drives itself — read, run, edit, re-run, answer — and Codex keeps looping until it sets end_turn = true. Each loop is one run_sampling_request.

Context once, security every time

The memory summary, AGENTS.md, and environment are assembled once at the top of the session. But every single tool call re-runs the approval + sandbox gauntlet.
