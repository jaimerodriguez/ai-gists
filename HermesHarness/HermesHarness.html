The life of a turn

One trip around the loop, in plain language — from the moment a message arrives (from a terminal, Telegram, an editor, or a cron job) to the moment Hermes answers. Everything after this section is the same story told in depth.

loop machinery gate / policy decision
1
A message arrives §1

From any of eight surfaces — interactive CLI, TUI, one-shot, batch, messaging gateway, editor (ACP), MCP bridge, or cron — every one of them funnels into the same method: AIAgent.run_conversation(...). One user message = one turn; a turn will internally make many API calls.

2
The turn prologue runs §3

build_turn_context() resets per-turn counters and mints a fresh iteration budget of 90 API calls, refreshes MCP connections, appends your message to history, restores (or builds) the system prompt, persists a crash-resilience snapshot, and runs a preflight compression check in case the conversation is already near the context limit.

3
The system prompt is assembled once, then frozen §2

Three ordered tiers — stable (identity from SOUL.md, tool guidance, skills index), context (AGENTS.md / CLAUDE.md files), volatile (memory snapshot, a date-only timestamp) — joined and cached for the whole session. Even "volatile" content is a frozen snapshot: the prompt stays byte-stable so provider prefix caches keep hitting.

4
The request payload is built §2

Cached system prompt + full message history, sanitized per provider. Anything dynamic — recalled memories, plugin context, MoA advice — is injected into the last user message, never the system prompt, to protect the cache prefix. Cache-control breakpoints are stamped on the system message and the last three messages.

5
The model is called — on a background thread §4, §8

The HTTP round-trip runs on a daemon worker thread so the main loop can poll for interrupts (Esc / a new message) without blocking. Streaming is always preferred — it enables 90-second stale-stream detection the blocking path lacks. A transport for the provider's api_mode (chat-completions, Anthropic, Responses, Bedrock) converts messages and normalizes the reply.

5aDeltas stream in; a stateful think-scrubber strips <think> blocks even when a tag is split across chunks.
5bPartial tool calls accumulate — names are assigned, arguments concatenated (some providers resend the full name every chunk).
5cOn failure, a 24-category error classifier decides: retry, compress, rotate credential, or fail over to another provider.
6
The fork: tool calls, or a final answer? §3

If the normalized response carries tool_calls, Hermes validates and repairs them (name resolution, JSON-argument repair, dedup), appends the assistant message, and dispatches the batch. If it's plain text, it heads for the exit — but has to pass the recovery and verification gates first (step 9).

7
Tools execute — up to 8 in parallel §5, §6

A parallel-safety analysis decides concurrency: read-only tools always parallelize; read_file/write_file/patch only when their paths don't overlap; clarify never. Workers run in a daemon thread pool with per-thread interrupt fan-out.

7aGuardrails check for repeated identical failures or no-progress loops before each call.
7bDangerous shell commands hit ~47 vetting regexes plus a stricter "hardline" tier that even yolo mode can't pass.
7cResults from the web, browser, and MCP servers are wrapped in <untrusted_tool_result> fences — data, not instructions.
7dOversized outputs are offloaded to sidecar files with a 1,500-char preview left inline; budgets scale with the model's context window.
8
Results append; the loop continues §3, §7

Tool results append in original call order as role:"tool" messages; the transcript flushes to SQLite before tool side effects so a self-destructive command can't lose history. Using the provider's real token count, a mid-loop compression check may summarize the middle of the conversation. Then: consume one unit of budget, go to step 5.

9
The exit gates: prove it before you claim it §6

A text-only reply doesn't automatically end the turn. An empty/thinking-only recovery ladder retries hollow responses; an "I'll do X" acknowledgment detector forces continuation; and if files were edited this turn, the verify-on-stop gate checks a SQLite evidence ledger — did a matching test/lint/build command pass since the last edit? If not, a synthetic nudge sends the model back to run verification (max 2 attempts).

10
The finalizer runs — no matter what §3

finalize_turn() handles every exit: if the budget ran out with no answer, it makes one extra tool-less API call asking the model to summarize progress; it closes interrupted tool sequences, saves the trajectory, persists the session, and computes token/cost accounting. Each cleanup step is independently guarded so a failure can't discard the response.

11
The answer is delivered §1

The result dict — final response, exit reason, usage, cost, cleanup errors if any — flows back to whichever frontend called: printed in the terminal, streamed over JSON-RPC to the TUI, sent to your Telegram chat, or returned to the editor.

12
After you've got your answer, Hermes studies §10

A background review fork — a whitelisted clone of the agent that can only call memory and skill tools — replays the turn and asks "did I learn anything worth keeping?" A session title generates asynchronously. Later, when idle, the Curator ages out unused agent-created skills and can consolidate them.

That's the whole machine. The rest of this page walks the same circle with the hood open — file names, line numbers, data structures, and the design decisions that make this loop different from other harnesses.

§ 01

Architecture: eight frontends, one engine

Everything hangs off one mental model: a single synchronous engine — AIAgent — whose run_conversation() method is the only door into the loop. Frontends differ wildly (REPL, React-in-terminal, chat platforms, editors, cron); the engine does not.

The engine lives in run_agent.py run_agent.py:393 — a ~6,000-line class holding LLM clients, the tool registry, and session state. Its run_conversation method run_agent.py:5745 is a thin forwarder to the free function that actually is the loop: agent.conversation_loop.run_conversation conversation_loop.py:518.

Confusingly for newcomers, cli.py (16k lines) does not define the agent — it's the interactive terminal frontend. Its AIAgent symbol cli.py:835 is a lazy proxy that imports the real class from run_agent on first use. Above both sits hermes_cli/main.py — the hermes command's argparse router with ~40 subcommands.

§ 03

The agent loop in detail

One function, three phases: a prologue that mutates the agent into turn-ready state, a while loop where each iteration is one API call, and a finalizer that runs on every exit path.

The loop guard

# conversation_loop.py:633 — the whole turn hangs on this line
while (api_call_count < agent.max_iterations
        and agent.iteration_budget.remaining > 0) or agent._budget_grace_call:
    ...
    if agent._budget_grace_call:
        agent._budget_grace_call = False
    elif not agent.iteration_budget.consume():
        _turn_exit_reason = "budget_exhausted"

max_iterations defaults to 90 agent_init.py:177. The IterationBudget iteration_budget.py:17 is a lock-guarded counter minted fresh each turn in the prologue turn_context.py:248 — subagents get their own (default 50), so a delegating turn can spend more than 90 calls in aggregate. Two details worth noting:

  • Refunds. Tool batches consisting solely of execute_code refund their budget unit conversation_loop.py:4708 — RPC-style code execution shouldn't consume "thinking" iterations.
  • A dormant grace flag. _budget_grace_call grants one over-budget call — but nothing in the current tree ever sets it True. It reads as an intentional extension point.

The prologue: build_turn_context()

Before iteration 1, turn_context.py:119 performs ~15 ordered steps: stdio guard, runtime binding, between-turns MCP refresh, retry-counter and budget reset, connection health check, appending the user message turn_context.py:307, restore-or-build of the system prompt, a crash-resilience DB persist, preflight compression (up to 3 passes if the conversation arrives over-threshold) turn_context.py:344–432, the pre_llm_call plugin hook, and recording the execution thread id for interrupt scoping. The returned TurnContext dataclass turn_context.py:92 carries only the locals the loop reads back — the real work is deliberate mutation of the agent.

Concurrency model, stated plainly

Threads, not asyncio. The loop is synchronous on one "execution thread." API calls run on daemon worker threads (each with its own HTTP client — interrupting closes only the worker-local client, avoiding an FD-recycling race chat_completion_helpers.py:210–240). Tools run on a daemon pool (§5). Interrupts are cooperative flags checked at iteration boundaries and inside the tool wait loop — plus a request-local cancel flag distinct from the turn flag, because gateway-cached agents can have worker threads that outlive a turn.

§ 04

The streaming state machine

The chat-completions SSE consumer is where Hermes's "works with everything" ambition earns its keep: the accumulator carries workarounds for a bestiary of provider quirks, each with the offending provider named in a comment.

§ 05

The tool system

A self-registering registry with live availability probes, a composable toolset algebra, two dispatch paths (parallel and sequential), context-scaled output budgets — and a "tool search" bridge so 90 MCP schemas don't have to ride in every request.

Registration & selection

Tools self-register at import time file_tools.py:2170 into a ToolRegistry whose entries carry the schema, handler, an emoji, a per-tool result-size cap — and two unusual fields: check_fn, a runtime availability probe ("is docker installed?") TTL-cached ~30s with a last-good grace window so a flaky probe can't yank a tool mid-session registry.py:145–196; and dynamic_schema_overrides, a callable that lets a schema's description reflect live config at definition-build time.

Toolsets are a static dict with recursive includes (composites like debugging = web + file), flattened with cycle detection toolsets.py:687, merged with plugin- and MCP-registered tools at resolution time. Schema assembly is memoized on a key that includes the registry's mutation generation and a config-file fingerprint model_tools.py:319.

Dispatch

All exceptions become {"error": sanitized} results; the loop never dies on a tool.

The Tool Search bridge

The clever part is dispatch: tool_call is unwrapped to the real underlying tool and re-dispatched recursively, so every guardrail, approval check, and hook sees the true tool name, and a per-agent scope cache stops restricted subagents from reaching ungranted tools through the bridge model_tools.py:1104–1138.

Skills & MCP, briefly

Skills are markdown instruction payloads (SKILL.md + YAML frontmatter) surfaced three ways: an index in the system prompt, three management tools (skills_list/skill_view/skill_manage), and slash-command invocation. Loading a skill substitutes template vars and can (opt-in) expand inline !`cmd` shell snippets with timeout and output caps skill_preprocessing.py:65–125. MCP runs both directions: as a client, discovered tools register under prefixed names with live tools/list_changed re-registration mcp_tool.py:1563; as a server, mcp_serve.py exposes Hermes's conversations (not its tools) to external MCP clients — read, send, poll events, answer permission prompts.

§ 06

Safety, guardrails & verify-on-stop

Four independent layers — command vetting, path guards, loop circuit-breakers, and prompt-injection framing — plus the most distinctive gate in the codebase: the turn cannot claim "done" after editing code without fresh verification evidence.

Layer 1 · Command vetting

~47 pre-compiled regexes catch destructive shell patterns (recursive rm, mkfs, dd, SQL DROP/DELETE-without-WHERE, fork bombs, PowerShell encoded commands) tools/approval.py:546–760. A stricter hardline tier exists that even --yolo mode can't pass, plus user deny rules and persisted approval allowlists.

Layer 2 · Path guards

file_safety.py — candidly documented as "defense-in-depth, NOT a security boundary" — deny-lists writes to credential files (~/.ssh, .netrc, /etc/sudoers, Hermes's own .env and OAuth stores) and blocks reads of any project-local .env* anywhere on disk, plus the skills-hub cache (a prompt-injection carrier) file_safety.py:98–304. Soft warnings (not blocks) cover cross-profile writes and sandbox mirrors.

Layer 3 · Loop guardrails

A per-turn controller hashes each call's canonical args (SHA-256 — no raw args stored) and tracks failure/no-progress signatures: exact-failure warn@2/block@5, same-tool warn@3/halt@8 tool_guardrails.py:63–124. Distinctively, warnings are on but hard-stops are opt-in — the default posture is "nudge the model, don't kill the turn." A block synthesizes a fake tool result explaining why.

Layer 4 · Untrusted-content framing

The verify-on-stop gate

The verify-on-stop gate A text response alone does not end a turn that edited code. Hermes keeps a passive SQLite evidence ledger (verification_evidence.db): every foreground terminal command is token-matched against the project's canonical verify commands — with equivalence expansion (pytest ≡ python -m pytest ≡ uv run pytest) — classified by kind (test/lint/typecheck/build) and scope, and recorded pass/fail by exit code verification_evidence.py:175–427. Editing a file marks the evidence stale. When the model tries to finish right after editing code with no fresh passing evidence, the loop injects a synthetic follow-up — run the verify command, read failures, repair — bounded to 2 attempts verification_stop.py:245–310. No canonical command? It's told to write an ad-hoc hermes-verify-* script, which the ledger recognizes as evidence.

The gate is surface-aware: on by default for coding surfaces (CLI/TUI/desktop), off for chat platforms where a verification narrative would be noise — default-deny for unknown new surfaces verification_stop.py:86–170. Doc-only edits never trigger it. A cousin mechanism in the finalizer prevents over-claiming: a file-mutation verifier footer compares claimed edits against tool results that actually landed (write_file must report bytes_written; patch must report success:true) tool_result_classification.py:12–26.

Two further safety systems are session-scoped rather than turn-scoped: checkpoints — the working directory is snapshotted before file-mutating tools and destructive terminal commands tool_executor.py:466–486 — and secret hygiene: redaction of ~30 vendor key patterns is on by default and snapshotted at import time, so a model-generated export HERMES_REDACT_SECRETS=false can't disable it mid-session redact.py:59–90.

§ 07

Context management & the cache discipline

A pluggable ContextEngine whose default implementation summarizes the middle of the conversation — triggered by real provider token counts, guarded against thrashing, and engineered never to break the cache prefix it doesn't have to.

When compression fires

# context_compressor.py:939–950 — threshold reserves output space, floors at 64k
effective_window = context_length - (max_tokens or 0)
pct_value = int(effective_window * threshold_percent)      # default 0.50
floored = max(pct_value, MINIMUM_CONTEXT_LENGTH)           # 64_000
if effective_window > 0 and floored >= effective_window:  # tiny local models
    return max(1, min(int(effective_window * 0.85), effective_window - 1))
return floored

Three trigger sites: the turn prologue (preflight, rough char/4 estimate), the pre-API pressure check (estimate including tool-schema bytes), and post-tool-batch using the provider's real prompt_tokens. Anti-thrash guards: a cooldown after summarizer failures, a defer-to-real-usage flag for one turn after each compaction (the rough estimate is known-noisy then), and a refusal to compress after two consecutive passes that saved <10% context_compressor.py:1136–1173.

What compression actually does

  1. Cheap pre-pass, no LLM: old tool-result bodies become one-line summaries ([terminal] ran `npm test` → exit 0, 47 lines); duplicate results dedup to the newest copy.
  2. Protect the head — which decays to system-prompt-only after the first compaction, so early conversation turns don't fossilize forever.
  3. Protect the tail by token budget, walking backward to ~20% of threshold, never cutting inside a tool-call/result group, always anchoring the last user and last visible assistant message.
  4. Summarize the middle with the auxiliary (cheap) model into a structured template whose first field is "Active Task — verbatim the last unfulfilled user input." Rules include secret redaction, answer-in-the-user's-language, and temporal anchoring — completed actions rewritten as dated past-tense facts so a resumed model doesn't re-execute them.
  5. Reassemble: head + summary + tail, with an explicit end-marker ("respond to the message below, not the summary above") so weak models don't treat the summary as fresh input context_compressor.py:2670–3045.

Failure policy is notable: auth and transient-network failures during summarization always abort — better to preserve the session than destroy the middle for a placeholder context_compressor.py:2826–2872. Historical images are stripped to placeholders so multi-MB base64 doesn't persist forever.

Prompt caching: one layout, four breakpoints

system_and_3 — the system prompt plus the last three non-system messages get cache_control markers prompt_caching.py:84–119, skipping messages a provider would silently ignore the marker on (a wasted breakpoint). Two transport layouts exist because OpenRouter needs the marker inside a content part while native Anthropic relocates a top-level marker. TTL is 5m by default, 1h opt-in (2× write cost, amortized over sessions with long pauses).

Also here @-references (@file:…, @diff, @staged, URLs) expand concurrently into the message, refused outright above 50% of the window context_references.py:161–197; and a live context-breakdown meter re-derives the prompt tiers to show per-category usage (system / tools / skills / MCP / memory / conversation) in the UIs.
§ 08

Providers & resilience

Twenty-nine bundled providers behind a two-tier abstraction: declarative ProviderProfiles (data) and four api_mode transports (bytes). The internal lingua franca is OpenAI chat-completions; everything else adapts.

A profile declares identity, auth, endpoints, and quirks — including where reasoning parameters go (Kimi: top-level; OpenRouter: extra_body) and an OMIT_TEMPERATURE sentinel for providers that reject the field providers/base.py:38–160. Profiles ship as plugin directories (plugins/model-providers/<name>/); user plugins override bundled ones. Transports own convert_messages → convert_tools → build_kwargs → normalize_response for one api_mode each — chat_completions, anthropic_messages, codex_responses, bedrock_converse — and explicitly do not own retries, streaming, or credentials, which stay on the agent transports/base.py:2–8.

The error classifier: 24 named failure modes

FailoverReason distinguishes, among others: rate_limit vs upstream_rate_limit (an aggregator's own 429 → rotate credential; an upstream model's 429 through OpenRouter → fail over, don't rotate), overloaded, context_overflow vs payload_too_large vs image_too_large, content_policy_blocked vs provider_policy_blocked, thinking_signature, and ssl_cert_verification — which fails fast, never retries error_classifier.py:24–72. Each classification carries action hints the retry loop obeys: retryable, should_compress, should_rotate_credential, should_fallback.

  • Reasoning-aware timeouts. § 10

Beyond the loop

The loop above is the request/response machinery. A set of subsystems built around it does verification gating, knowledge accumulation and maintenance, multi-model turns, and offline data generation — each covered in depth, with citations, on a companion page.

The short list:

  • Verify-on-stop with an evidence ledger — the turn can't claim "done" after edits without fresh passing verification recorded in SQLite (§6).
  • A post-turn self-improvement fork — a tool-whitelisted background clone replays the turn and autonomously saves skills/memories.
  • The Curator — idle-time, deterministic-plus-LLM maintenance of the learned skill library: stale→archive lifecycles and "umbrella-building" consolidation.
  • A learning graph — skills + memory cards as a navigable knowledge graph with derived edges, rendered as a constellation (desktop) or timeline (terminal).
  • Mixture-of-Agents turns — parallel advisor models + an aggregator inside the normal tool loop, with per-advisor cost attribution and full traces.
  • Multimodal provider registries — image/video/TTS/transcription/search/browser as uniform, hot-swappable plugin backends.
  • Multi-tenant secret scoping — fail-closed ContextVar secret isolation for multiplexed profiles; import-time-locked redaction; streaming think-scrubbing.
  • A training-data factory — batch runners, Docker/Modal sandboxes, trajectory compression, and eval loops built into the same repo.
  • An animated pet — a sprite mascot whose animation mirrors live loop state (tool running → RUN, all todos done → JUMP), generatable via image models.

§ 11

Transferable patterns

Design moves worth stealing for any harness, independent of Python or Hermes's feature set.

P-01

Byte-stable prompts or bust

Build the system prompt once, freeze even "volatile" content as snapshots, use date-only timestamps, inject all dynamic context into the last user message, canonicalize JSON. Every decision defers to the cache prefix.

P-02

Budget, don't just cap

An explicit IterationBudget object — consumable, refundable, freshly minted per turn, separately allocated to subagents — beats a bare loop counter. Refunds let cheap RPC-style calls not count as "thinking."

P-03

Exit reasons are an enum, not a vibe

Every way a turn can end sets a named _turn_exit_reason. Post-mortems, UX explainers, and tests all read one string instead of reverse-engineering state.

P-04

Flush before side effects

Persist the transcript to the DB before executing tools. A tool that restarts the process — legitimately — cannot destroy the record of asking for it.

P-05

Force a summary on budget exhaustion

When iterations run out, one extra tool-less call — "summarize progress, no more tools" — converts a timeout into a progress report.

P-06

Classify errors into actions, not categories

The classifier's output is retryable / should_compress / should_rotate_credential / should_fallback hint flags. The retry loop obeys hints; it never re-derives meaning from strings.

P-07

Assign names, concatenate arguments

In streaming tool-call accumulation, the name field is atomic (some providers resend it whole every chunk); only arguments concatenate. One character (= vs +=) between correctness and read_fileread_file.

P-08

Verification is evidence, not assertion

Observe which commands actually ran and passed (a passive ledger); gate "done" on fresh evidence newer than the last edit. The model's claim of testing is not the same as a recorded exit code 0.

P-09

Wrap untrusted content at the seam

Web/browser/MCP results get delimiter-fenced "this is DATA" framing applied at the single choke point where tool results become messages — with forged-delimiter neutralization.

P-10

Scale output budgets to the window

Truncation caps as a fraction of the active model's context window (with floors), plus offload-to-file with inline previews — instead of one hardcoded constant that's wrong at both extremes.

P-11

Availability is a probe, not a list

Each tool carries a check_fn runtime probe, TTL-cached with a last-good grace window — tools appear and disappear with the environment, without flapping mid-session.

P-12

Nudge before you halt

Guardrails default to appending warnings to the transcript; hard-stops are opt-in. The model usually course-corrects when told it's looping — killing the turn is the last resort, not the first.

How this was researched

Static analysis of the full source tree at commit 18e840469, cross-checked by parallel subsystem reads and real request dumps captured from the wire.

Six independent deep-reads covered the core loop, prompt/context management, the tool system, the provider layer, frontends/persistence, and the distinctive-features inventory — each claim resolved to a file:line citation against the working tree, then reconciled into this document. Constants quoted (90 iterations, 8 workers, 50% threshold, 64k floor, 4 breakpoints, 29 providers) are the shipped defaults; most are config-overridable.

The wire evidence comes from Hermes's own debug facility: HERMES_DUMP_REQUESTS=1 writes a secret-redacted snapshot of each outbound request body to the sessions directory before the call fires agent_runtime_helpers.py:1377–1458. Three same-day sessions were examined: a six-round-trip coding turn and a parallel-write turn (claude-sonnet-5, responses inferred from request growth — each dump embeds the previous round's results), plus a deliberately traced session (claude-opus-4-8 over Claude Code OAuth) that surfaced the mcp__ masquerade layout and, when the account hit its usage cap, the classifier's non_retryable_client_error path — four differently-shaped retry attempts before giving up, exactly as §8 predicts. One honest caveat: the dump's URL field prints /chat/completions for every api_mode; the anthropic_messages transport actually posts to /v1/messages.
