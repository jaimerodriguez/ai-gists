Writing Evals for AI Agents

Aggregated best practices for evaluating tool-using agents — and the concrete design behind the agent-evals project, which compares the same research/Q&A agent built on the Claude Agent SDK and on LangChain.

1The mental model: an eval is four things

An agent eval is not "a benchmark." It is a small system with four parts, and most eval failures are a failure of exactly one of them:

  • Task — a question or job with a knowable right answer (or right behavior, e.g. abstaining).
  • Environment — the tools and state the agent acts in. For evals it should be hermetic: local, deterministic, resettable.
  • Grader — code checks and/or an LLM judge that turn a transcript into pass/fail (or a score).
  • Harness — the runner that executes agents × cases × trials, persists every transcript, and computes statistics.

What makes agent evals different from classic single-turn LLM evals: the unit of evaluation is a run (many model calls, tool calls, and intermediate states), not a single completion. That means two extra judgment surfaces — the trajectory (what the agent did) and the end state (what is true afterwards) — plus much higher variance, which forces repeated trials and real statistics (§6).

2Grade outcomes; observe trajectories

The single most load-bearing principle in the field right now:

Core principle

Grade primarily on the verified end state — what is actually true in the environment or in the final answer — never on the agent's own claim that it succeeded. And don't hard-code the expected step sequence: agents regularly find valid approaches the eval designer didn't anticipate, so exact-order checks produce brittle evals. verified 3–0

Anthropic, Demystifying evals for AI agents (Jan 2026).

Trajectory evaluation is still valuable — as a complement. Which tool the agent picks, how many calls it burns, whether it consults sources before answering: these behaviors only emerge when a real model runs end-to-end, and they explain why an outcome failed. The practical guidance is to use flexible trajectory matchers — unordered, subset, superset — rather than strict order matching, exactly the modes LangChain's agentevals package ships. verified 3–0 ×5

In the project this shows up as: the LLM judge grades the final answer against a reference (outcome), while code graders check "expected tools were used," "the KB was consulted at all," and "call count within budget" (trajectory) — as separate, named signals that never get averaged into mush.

3Bootstrapping a dataset from zero

Start with 20–50 hand-written tasks. Anthropic's guidance is explicit: 20–50 simple tasks drawn from real (or realistically anticipated) failures is a great start, and the quality bar for each task is the two-expert rule — a good task is one where two domain experts would independently reach the same pass/fail verdict. verified 3–0 If reasonable people could argue about whether an answer passes, the case will generate judge noise forever; sharpen the expected answer until the verdict is obvious.

Cover behavior types, not just topics. The 21-case dataset deliberately spans:

  • single_hop — one lookup; establishes a floor;
  • multi_hop — facts that live in two documents; tests planning and synthesis;
  • calculation — forces correct tool choice (the calculate tool) and verifiable arithmetic;
  • abstention — questions the corpus can't answer. The right behavior is refusing to invent; this is the category that catches hallucination, and most homemade eval suites forget it entirely.

Split capability from regression. Maintain two suites: capability evals start at low pass rates (a hill to climb as you improve the agent), regression evals sit near 100% and alarm on backsliding. When a capability eval saturates — passing every trial for every variant for weeks — it graduates into the regression suite and you write harder cases. verified 3–0

Use invented facts. The corpus describes a fictional company, so a correct answer is proof the agent used the tools — the facts cannot exist in pretraining data. This is the cheapest available defense against contamination, and it makes "answered from memory without touching the KB" a mechanically detectable failure.

LLM-seeded cases: seed, review, promote

Using a strong model to generate candidate cases from your corpus is a legitimate productivity lever — but the human review step is non-negotiable. LLM-generated evaluators and eval assets inherit all the problems of the LLMs they evaluate, and require further human validation. verified 3–0 The project's scripts/seed_dataset.py encodes the safe workflow: generate to candidate_cases.jsonl → human verifies each expected answer against the corpus → only then append to the dataset. Skipping the review makes your eval circular: the generator's mistakes become "ground truth," and a same-family judge will happily agree with them.

A dataset-design pattern worth copying (from DeepEval): a case (a "golden") stores only the input and expected results; everything runtime-generated — actual output, tool calls, retrieved context — is populated at evaluation time. That separation is exactly what lets one dataset drive two different agent frameworks. verified 3–0

4The grading hierarchy: code first, judge second

Rule of thumb

Prefer deterministic, code-based graders wherever a clear rule can be written; clearly defined rules plus an explanation of which rule failed is the best form of verification feedback. Reserve the LLM judge for what code genuinely cannot check. verified 3–0

Reasons: code graders are free, instant, perfectly reproducible, and cannot drift. Every check you move from judge to code removes a noise source and a cost line. In practice a healthy stack looks like:

LayerChecksCost
1. Code gradersrequired facts/numbers present (normalized), expected tools called, KB consulted at all, call count ≤ budget, run completed without errorfree
2. Pointwise judgesemantic correctness vs. reference, groundedness (no invented facts), abstention quality1 judge call per trial
3. Pairwise judgeA-vs-B preference on the same question, position-debiased2 judge calls per pair; run on demand

Note what's not here: string-equality on the whole answer (too brittle for free-form text) and embedding similarity (too forgiving — a fluent wrong answer scores high). Substring checks on the load-bearing numbers, with currency/comma normalization, hit the sweet spot for a Q&A task.

5LLM-as-judge, properly

5.1 The philosophy

A judge is not an oracle; it is another model with its own failure modes, deployed in a measurement role. Two verified findings anchor everything else:

  • Judges inherit the failure modes of the models they evaluate and must be calibrated against human grading before being trusted. verified 3–0
  • Criteria drift is real: you need criteria to grade outputs, but grading outputs is what teaches you the criteria — some criteria only become visible once you've seen real model outputs. Build the harness expecting rubrics to be revised as you read transcripts; don't treat the first rubric as fixed. verified 3–0 (Shankar et al., Who Validates the Validators?, UIST 2024)

The practical calibration loop (the "EvalGen pattern," from the same paper): have a human grade a subset of outputs, then select/tune the judge prompt (or code assertion) that best agrees with the human grades. verified 3–0 Concretely: after the first eval run, read ~20 judge verdicts next to the transcripts, count disagreements with your own judgment, and edit the judge prompt until agreement is high. Only then do the aggregate numbers mean anything. Re-calibrate whenever you change the judge prompt or model — and version both.

5.3 Known biases and the standard mitigations

The bias catalog below is standard literature (MT-Bench / Zheng et al. 2023 and follow-ups) — well established, though not independently re-verified in this research pass.

  • Position bias — pairwise judges favor one slot (often the first). Mitigation: judge each pair twice with positions swapped; count a winner only when both orderings agree, else record a tie. The project's judge_pair() does exactly this and reports how many verdicts flipped — a built-in bias meter.
  • Verbosity bias — longer, more polished answers win undeservedly. Mitigation: instruct the judge to grade substance against the reference facts, and say explicitly that length and polish don't count. Reference-anchored grading largely defuses this.
  • Self-preference bias — models favor their own family's outputs. In this setup both agents run Claude, so it can't tilt A vs. B — but it can inflate absolute scores. Mitigation: keep the judge model independent and configurable; if numbers look suspiciously rosy, re-judge a sample with a different-family model and compare.
  • Sycophancy toward confident tone — assertive wrong answers pass. Mitigation: the rubric ties passed to enumerated key facts, not to overall impression.

5.4 Judge prompt design — writing the instructions the judge model receives

A disambiguation first, because the phrase reads two ways: this section is about designing the prompt you give to the judge — the grading instructions the judge model runs with — not about judging prompts. The judge never grades a prompt; it grades transcripts. (Your agent's own prompt does get evaluated, but only indirectly: you change it, re-run the eval, and compare scores. The eval is how prompt changes get judged; the judge itself only ever sees answers.)

Why this deserves its own section: the judge prompt is your measurement instrument. Every weakness in it — a vague rubric, a missing reference, an unparseable output format — becomes noise in every number the eval produces. A judge call is an ordinary LLM call with three parts, and judge prompt design is the craft of getting all three right:

  1. The grading instructions (system prompt) — who the judge is, the rubric, and an explicit rule for every grade it can hand out;
  2. The evidence (user message) — everything about this one trial: the question, the reference answer and key facts, and the agent's answer plus its tool calls;
  3. The output contract (a JSON schema) — the exact fields the verdict must come back in.

The shape, annotated from the project's judge_run() in graders/llm_judge.py:

SYSTEM — grading instructions (the rubric)
  "You are grading the answer of a research agent..."
  correctness = "correct" only if every key fact is present and accurate
  grounded = false if the answer asserts any fact the reference doesn't support
  if MUST_ABSTAIN: fabricating an answer is an automatic fail
  passed = correct AND grounded (or an appropriate abstention)
  "Write your analysis first, then the verdict fields."

USER — the evidence for this one trial
  QUESTION / REFERENCE ANSWER / KEY FACTS REQUIRED / MUST_ABSTAIN
  AGENT'S TOOL CALLS   (the trajectory)
  AGENT'S FINAL ANSWER

OUTPUT — enforced JSON schema (pydantic)
  { analysis, correctness, grounded, abstained_appropriately, passed }

Five recommendations follow. Each is stated the same way: the thing to do, the failure it prevents, and where it lives in the project's code — every bolded item is a desired practice, not a failure mode.

  • Anchor to a reference.
    Do Give the judge the case's expected answer and enumerated key facts, and instruct it to grade strictly against them (here, it's also told the corpus is fictional).
    Prevents The judge grading from its own world knowledge — empty for a fictional corpus, stale for a real one — so the verdict measures the judge's knowledge instead of the agent's answer. Handing over the answer key also converts a hard judgment ("is this answer good?") into an easy one ("does it match the key?"), and judges are far more reliable at comparison than at open-ended assessment.
    Where judge_run() puts REFERENCE ANSWER and KEY FACTS REQUIRED in the evidence; the instructions open with "grade strictly against the reference."
  • Structured output.
    Do Force the verdict into a JSON schema (pydantic model + the SDK's messages.parse) so every field comes back as one of the allowed values.
    Prevents Prose verdicts ("I'd say this mostly passes, though...") that you parse with regexes — every parse failure or invented label becomes silent grading noise.
    Where The JudgeVerdict model in llm_judge.py.
  • Analysis before verdict.
    Do Order the schema so the free-text analysis field comes before the grade fields — the model generates output in field order, so this makes it lay out the evidence before committing to grades.
    Prevents A snap verdict followed by a rationalization of it, which is what a verdict-first schema invites.
    Where analysis is the first field of JudgeVerdict; it's also what you read during calibration (§5.1) to see why the judge graded as it did.
  • Per-dimension fields with an escape hatch.
    Do Grade each dimension in its own field — correctness (3-level, so partial credit exists), grounded, abstained_appropriately — and make inapplicable dimensions nullable (the "Unknown" escape hatch). Structured per-dimension rubrics with an escape hatch and partial credit measurably improve judge reliability. verified 3–0
    Prevents Holistic scores ("7/10") that confound failures — was it wrong, ungrounded, or just wordy? — plus forced grades on dimensions that don't apply (pure noise), and the cliff where a 95%-right answer scores like an empty one.
    Where The JudgeVerdict fields; abstained_appropriately is nullable.
  • A small, explicit pass rule.
    Do Define passed in the instructions as a stated formula over the dimension grades ("passed = correct AND grounded, or an appropriate abstention") so the judge applies a rule.
    Prevents Impression-based pass/fail that drifts from run to run.
    Where The "passed = true only if..." line in the judge's system prompt — and because the rule is one line of text, it's versionable, auditable, and exactly the line you edit when calibration (§5.1) shows the judge is too strict or too lenient.

Two refinements to reach for as the suite matures, both extensions of the same idea — make the judge's job smaller and more mechanical: few-shot graded examples (paste two or three example answers with their correct verdicts into the instructions; the cheapest big accuracy lever for fussy rubrics) and one judge call per dimension (split dimensions into separate calls if grading one seems to bleed into another).

5.5 Judge-to-seed vs. judge-to-grade

These are two distinct, both-legitimate roles with different safety rails:

  • Seeding (generation time): the model drafts cases/expected answers → human review is the quality gate → reviewed cases become ground truth. The model's judgment is advisory.
  • Grading (evaluation time): the model applies your rubric to transcripts → calibration against human spot-checks is the quality gate → verdicts become metrics. The model's judgment is the measurement instrument.

The failure mode to avoid is letting the same unreviewed model output serve both roles — generating the answer key and then grading against it. Keep a human checkpoint between the two.

6Nondeterminism and statistics

Agent runs are nondeterministic — and on current Claude models (Opus 4.7+/Sonnet 5) you cannot even set temperature; the parameter is rejected. Determinism is therefore approximated by repetition plus statistics, not seeds.

First, what k is. k is simply the number of times you re-run each test case — the trials setting in the project's config.json (default 3). "k trials per case" with k=3 means: ask case kb-014 three separate times and record pass/fail each time. There is no magic value; the practical guidance is k=1 for quick iteration (accepting that you can't tell skill from luck), k=3–5 for normal development (enough to see instability), and k=10+ when you need to defend a reliability claim. Cost scales linearly with k.

  • Run k trials per case from a clean, isolated environment so failures aren't correlated through shared state. verified 3–0 (The task is stateless-per-run, which gives isolation for free.)
  • pass@k vs. pass^k — two different questions asked of the same k trials. pass@k is the probability that at least one of the k trials succeeds ("can it do this?"); pass^k is the probability that all k succeed ("does it do this every time?"). verified 3–0 The distinction feels obvious once seen — the names exist because they're standard vocabulary (pass@k from OpenAI's HumanEval code benchmark, pass^k from METR's agent-reliability work), which lets your numbers be compared with published ones. Worked example: a case with a 90% per-trial success rate at k=3 scores pass@3 = 1 − 0.1³ = 99.9% but pass^3 = 0.9³ = 72.9% — same agent, same trials, wildly different numbers. Choose by whether a retry exists in real usage: pass@k when someone can pick the good attempt (code filtered by a test suite, a human reviewing drafts); pass^k when the agent acts autonomously and every run must be right — which is why ship decisions should weight pass^k. (Harness detail: the report shows pass@1, the plain average pass rate over all trials, alongside pass^k, the fraction of cases that passed every trial.)
  • Paired bootstrap confidence interval (CI) on the difference. The question this answers: agent A scores 76%, agent B scores 69% — is that +7-point gap real, or the luck of these particular ~20 questions? Unpacking the name, one word at a time:
    • Confidence interval — instead of the single number (+7), you get the range of gaps that are plausible given your data: "somewhere between X and Y."
    • Bootstrap — how that range is computed: simulate re-running the experiment thousands of times by resampling your cases, recomputing the gap each time. The middle 95% of the simulated gaps is the 95% CI.
    • Paired — each simulated re-run keeps a question's A-score and B-score together, because both agents answered the same questions. A hard question drags both agents down equally, so question difficulty stops being a source of noise.
    How to read it: a CI that straddles zero (say, −2 to +16 points) means a zero gap is plausible — the difference is not established; decide on cost/latency instead, or add cases. A CI that excludes zero (say, +3 to +19) means the gap is real at this sample size. (Adding cases narrows the CI faster than adding trials.)
  • Read the discordant (disagreement) cases. The cases where the two agents disagree — one passes, the other fails — are worth more than any aggregate: they localize the actual behavioral difference. The report lists them by ID so you can pull the transcripts.

7Fair A/B comparison of two agent frameworks

The question "which framework is better?" is only meaningful once everything except the framework is held equal. The confounds, in the order they usually bite:

  1. Model. Same model on both sides isolates the framework effect (the scaffold: prompting harness, loop mechanics, context management). Different models per side measures end-to-end stacks — also valid, but a different experiment. The project's config.json makes this an explicit switch.
  2. Tool surface. The Claude Agent SDK ships built-in tools (WebSearch, WebFetch, file I/O, Bash) while LangChain tools are wired manually — a real asymmetry the design must normalize. verified 3–0 The SDK adapter passes tools=[] to strip every built-in and exposes only the shared MCP tools, so both agents see byte-identical tool names, schemas, and descriptions.
  3. System prompt. Defined once in task.py, injected into both.
  4. Environment flakiness. Hermetic local tools; a network hiccup can otherwise hand either side unearned failures.
  5. Grader symmetry. One shared grading stack, blind to which framework produced the transcript (adapters normalize tool names before grading).

Then report quality and efficiency side by side: pass@1, pass^k, tokens, latency, tool calls, and cost per trial. Frameworks that tie on correctness routinely differ 2× on tokens or latency — that difference is a finding, not a footnote.

Honest caveat

What this experiment measures is each framework's scaffold on this task at this config. It does not measure developer experience, ecosystem, observability, or how each framework scales to bigger agents — judge those separately. And note the Agent SDK runs Claude Code's harness with its own system prompt around yours, so "same model" is still not "same prompt tokens." That's precisely the framework effect you're measuring — just name it when you present results.

8Pitfalls that quietly ruin eval programs

  • Eval overfitting / Goodhart. Iterating your agent prompt against the same 21 cases will eventually tune to the cases, not the capability. Hold out a few cases you never look at during development, and refresh capability cases as they saturate (§3).
  • Contamination. Public benchmark questions leak into training data. Invented-fact corpora, private test sets, and post-cutoff data are the defenses.
  • Judge drift. Changing the judge model or prompt silently re-scores history. Pin the judge model in config, version the prompt, and re-run a reference batch after any change before comparing new numbers to old ones.
  • Flaky tools. A tool timeout counts as an agent failure unless you separate infra errors from behavior errors. The records keep error distinct from a failed grade, and hermetic tools minimize the category.
  • Sandboxing. Agents with Bash/file/web tools must run in isolated environments during evals — both for safety and so trials can't contaminate each other via shared state. This task avoids it by having no side-effectful tools; the moment you add them, add per-trial sandboxes too.
  • Self-verification belongs in the agent too. Anthropic's gather-context → take-action → verify-work loop: agents that check their own output are more reliable — evals measure quality, but in-agent verification creates it. verified 3–0

9Tooling landscape (and why the project is code-first)

  • Code-first / build-your-own (the project). ~600 lines, zero platform dependencies, every judgment inspectable. The right call for learning the fundamentals and for a two-framework bake-off.
  • openevals + agentevals (LangChain, MIT, work fully without LangSmith): prebuilt LLM-as-judge prompts, exact-match/structured output evaluators, and — the interesting one — trajectory evaluators with four deterministic match modes (strict/unordered/subset/superset) plus reference-free LLM trajectory judging. They accept OpenAI-style message lists, so both agents' trajectories can be fed to them. A natural next dependency if you outgrow the built-in matchers. verified 3–0 ×5
  • DeepEval / promptfoo — OSS harnesses with metric libraries and CI integration; DeepEval's "golden" dataset model matches the project's schema (§3).
  • Hosted platforms (LangSmith, Braintrust, Langfuse, Arize Phoenix, W&B Weave) — add tracing UIs, dataset versioning, human-annotation queues, and online (production) evaluation. Worth adopting when the bottleneck becomes reviewing transcripts rather than running them.
