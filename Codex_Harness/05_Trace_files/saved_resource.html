03 The surprise in the code it wrote

Asked for "a couple asynchronous calls," a naive guess is tokio. Instead — because it had probed the sandbox and found no network — the model hand-rolled its own block_on executor with a no-op Waker, keeping [dependencies] empty. A real adaptation to the environment, not a canned answer.

use std::future::Future;
use std::task::{Context, Poll, RawWaker, RawWakerVTable, Waker};

fn main() { block_on(run()); }

async fn run() {
    let greeting = fetch_greeting().await;   // async call #1
    let audience = fetch_audience().await;   // async call #2
    println!("{greeting}, {audience}!");
}

async fn fetch_greeting() -> &'static str { "Hello" }
async fn fetch_audience() -> &'static str { "async Rust" }

fn block_on<F: Future>(future: F) -> F::Output {
    let waker = noop_waker();
    let mut context = Context::from_waker(&waker);
    let mut future = Box::pin(future);
    loop {
        match Pin::new(&mut future).poll(&mut context) {
            Poll::Ready(value) => return value,
            Poll::Pending => std::thread::yield_now(),
        }
    }
}   // + a noop RawWakerVTable (omitted)

04 The thing nobody predicted · a second thread

The internal log carried two thread IDs. The main one did the task. A separate thread ran in parallel making its own 7 model round-trips — entirely inside ~/.codex/memories/.

Second thread · the memory subsystem, live

It ran commands like sed MEMORY.md, sed memory_summary.md, find rollout_summaries/, and sed extensions/ad_hoc/instructions.md. Its spans were dominated by memories (106 lines), memory_summary (37), and — tellingly — write (12) and ad_hoc (1). Its own UserInput read: "## Memory Writing Agent: Phase 2 (Consolidation)…", and the code names it exactly — session source SessionSource::Internal(InternalSessionSource::MemoryConsolidation) (client.rs:712), with its own consolidation_model config. So it's a first-class memory-consolidation agent — a separate internal session that folds raw memories + rollout summaries into your file store. Confirmed later: it runs periodically, not every turn — it spawned alongside turn 1 but not during the turn-2 resume.

The internal log showed resume_thread_with_history (history reconstructed from the rollout) and no full-context re-injection — the only full_context hits were the full_context_window_limit token field. Turn 1's entire trace was replayed into the prompt, so the model skipped recon entirely and went straight to the edit: "I'll move the async workflow directly into main() and remove the block_on helper." Its whole turn was edit → cargo fmt/run → git diff → report.

It did exactly what was asked — deleted the block_on helper and inlined the poll loop into main(), still dependency-free (so nothing hit the sandbox):

fn main() {
    let waker = noop_waker();
    let mut context = Context::from_waker(&waker);
    let mut workflow = Box::pin(async {
        let greeting = fetch_greeting().await;
        let audience = fetch_audience().await;
        println!("{greeting}, {audience}!");
    });
    loop {                                    // the blocking now lives in main()
        match Pin::new(&mut workflow).poll(&mut context) {
            Poll::Ready(()) => break,
            Poll::Pending => std::thread::yield_now(),
        }
    }
}
