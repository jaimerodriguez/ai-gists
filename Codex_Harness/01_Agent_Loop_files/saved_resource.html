Codex Agent Harness — Architecture Map

Codex-rs · agent harness

The Codex agent loop, and the files that run it

A walkthrough of OpenAI Codex's Rust harness (codex-rs/). Every frontend speaks one language to a single core: send an Op down, receive a stream of Events back. Inside codex-core, a turn loop drives the model, dispatches tool calls through a sandbox-and-approval gauntlet, and feeds results back until the model stops.

~115 crates in workspace core harness: codex-core entry binary: codex (cli) wire types: codex-protocol

01 The whole system in one breath

Four frontends, one core. They never touch the model or the filesystem directly — they submit Op commands and render Event notifications. This uniformity is the single most important idea in the codebase.

02 Anatomy of a turn — the harness loop

A UserInput Op enters a submission queue, becomes a SessionTask, and runs run_turn. That function contains the loop that is the agent: request the model, stream its output, run any tool calls it asked for, feed the outputs back, and repeat until the model ends its turn.

05 Subsystems worth knowing exist

Not on the critical path, but they explain behavior you'll wonder about.

Context

History & compaction

Long sessions overflow the model's window. The harness measures token budget after every sampling request and, when needed, compacts history mid-turn and continues.

  • → context_manager/ — the model-visible history
  • → compact.rs, tasks/compact.rs — summarization
  • → turn.rs:917 — run_auto_compact
Approvals

The escalation ladder

AskForApproval sets the posture: Never → prompts become rejections; UnlessTrusted → only known-safe reads auto-run; OnRequest (default) → Guardian may auto-resolve; Granular → per-category toggles.

  • → protocol/approvals.rs
  • → guardian/review.rs — the auto-reviewer
  • → network_policy_decision.rs
Sandbox

Per-OS enforcement

Auto-approved commands still run inside a sandbox. The same shipped binary re-execs as the sandbox helper via arg0 dispatch.

  • → sandboxing/seatbelt.rs — macOS (.sbpl)
  • → linux-sandbox/, bwrap/ — Linux
  • → windows-sandbox-rs/ — restricted token
MCP

Tools in both directions

Codex is both an MCP client (calling out to external tool servers) and an MCP server (other clients drive Codex). Exposure control decides which external tools the model sees directly vs. via tool-search.

  • → codex-mcp/ — the client library
  • → mcp-server/ — Codex-as-server
  • → mcp_tool_exposure.rs
Multi-agent

Sub-agents & delegation

Codex can spawn collaborating sub-agents with distinct roles. The collaboration tools (spawn / wait / send) are registered like any other tool.

  • → core/src/agent/ — roles, registry, control
  • → tools/handlers/multi_agents*
  • → codex_delegate.rs
Persistence

Rollout & resume

Every turn is journaled so a session can be reconstructed, forked, or resumed. This is what codex resume and cloud tasks are built on.

  • → core/src/rollout.rs, rollout/ crate
  • → thread_store.rs, state/
