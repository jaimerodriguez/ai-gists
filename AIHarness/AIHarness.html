Author: Codex
Date: 2026-07-09
Scope: LLM agent harnesses that can use tools, manage context, persist work, and act with bounded autonomy.

Abstract

An AI harness is the deterministic runtime around a model. The model decides, but the harness assembles context, exposes tools, enforces permissions, records state, runs verification, and decides when a turn is complete. The loop itself is simple: call the model, execute requested tools, append observations, repeat until the model produces a final answer or the harness stops it. The hard part is everything around that loop.

This paper answers four questions:

  1. What components are required for a serious harness?
  2. What components are optional, conditional, or profile-specific?
  3. What memory and profile abstractions are useful?
  4. In what order should components call one another during a session, a turn, a tool call, and a subagent delegation?

1. What an AI Harness Is

An AI harness is the runtime substrate that converts a user's goal into a sequence of model calls, tool calls, state updates, and safety decisions. It is not the model, not the frontend, and not the tool implementation alone. It is the orchestration layer that controls:

  • What the model sees.
  • What the model can call.
  • What code or external systems are allowed to do.
  • How results are fed back into the model.
  • How work is recorded, resumed, summarized, verified, and audited.

The core lesson: the model is a decision component; the harness is the operating system for that decision component.

2. Required Components

The required set depends on whether the agent can act. A read-only chat assistant can skip several controls. A harness that runs tools, edits files, calls APIs, or spends money cannot.

2.1 Agent Loop

Required for any agentic harness.

The irreducible loop is:

append user message
while budget remains:
  response = model(context, tools)
  append response
  if response has no tool calls:
    break
  for each tool call:
    result = execute_tool(call)
    append result
return final response or controlled exit

The production version needs:

  • A max-iteration budget.
  • A cost/token budget.
  • An explicit exit reason enum such as final_response, budget_exhausted, interrupted, approval_denied, guardrail_halt, or error.
  • A finalizer that runs on every exit path.
  • [Edit: Added] Interrupts and steering. 

Why it matters: without budgets, denial-of-wallet and infinite retries become normal failure modes. OWASP explicitly calls out unbounded loops and token/cost tracking as security concerns.

[Edit pass: Note] This irreducible loop feels like a turn/task. To accomplish a goal, there should be an outer loop iterating the tasks, saving state, etc. 

2.2 Context Assembler

Required.

The context assembler decides what gets sent to the model on each call. It should separate content by stability and authority:

  1. Stable system/developer instructions.
  2. Project instructions.
  3. Profile/user preferences.
  4. Memory snapshots and recalled facts.
  5. Current user message and task state.
  6. Prior messages, tool calls, and tool results.
  7. Tool schemas.

[Edit pass: Bolded] The assembler must preserve prompt-cache discipline. OpenAI's prompt caching docs state that cache hits depend on exact prompt prefixes, so static content should go at the beginning and variable user-specific content at the end. Hermes and Codex both exploit that principle: stable instructions stay byte-stable; dynamic data rides late in the request.

Required behavior:

  • Keep the system prompt stable during a session.
  • Put volatile per-turn material near the end.  [Edit pass: Added] Sort convent by volatility.  
  • Canonicalize tool schemas and structured content where possible.
  • Include only the tools visible for this request.
  • Treat external content as data, not instructions.
  • [Edit pass: Added]. Keep prompts at right altitude. With newer models, you need less instructions. Provide just enough information to outline the desired behaviors and outcomes. 

2.3 Tool Registry and Dispatch Funnel

Required for acting agents.

A tool needs four things:

  • Name.
  • Description.
  • Input schema.
  • Execution function.

OpenAI's function calling guide describes the loop: make a request with available tools, receive a tool call, execute application-side code, return the tool output, and call the model again. The harness must own the execution step.

The registry answers: "What tools exist and are visible for this model, task, profile, and trust level?"

The dispatch funnel answers: "Given a specific tool call, what checks and wrappers run before and after the handler?"

Required dispatch steps:

  1. Parse and validate the call.
  2. Record the requested call.
  3. Apply pre-tool hooks or middleware.
  4. Consult permissions and approval policy.
  5. Acquire credentials, if needed.
  6. Execute in the sandbox or external tool boundary.
  7. Normalize result or error.
  8. Truncate or offload large output.
  9. Fence untrusted data.
  10. Record the result.

[Edit pass: Bolded]. All tools should pass through one funnel. Otherwise policy, telemetry, redaction, and error shaping will drift between handlers.

2.4 Permissions, Approval Policy, and Sandbox

Required for acting agents.

[Edit pass: Bolded] Separate two axes:

  • Permission or sandbox policy: what a process or tool may technically touch.
  • Approval policy: when the harness must stop and ask before doing something.

OpenAI's Codex sandboxing docs make the same distinction: the sandbox defines the technical boundary; approval policy decides when to ask before crossing it. OWASP adds least privilege, per-tool scoping, explicit approval for sensitive operations, and fail-closed behavior for policy failures.

Required controls:

  • Filesystem boundary.
  • Network boundary.
  • Tool allow/deny lists.
  • High-impact action classification.
  • Approval records bound to exact action parameters.
  • Deny-first precedence.
  • Audit trail.
  • Fail-closed default.  [Edit pass: Added] If risk classification, approval validation, policy lookup, or audit logging fails

Credentials are not permissions. Credentials should be retrieved only after policy approves an action, scoped to the specific tool/action, and injected into the executor or API client, not into the prompt.

2.5 Persistence and Resume

Required.

The harness needs an append-only record of:

  • User messages.
  • Model responses.
  • Tool calls.
  • Tool outputs.
  • Approval decisions.
  • Context compactions.
  • Verification evidence.
  • Exit reasons and usage.

Codex uses append-only rollout records as canonical history in the local trace. Hermes flushes transcript state before tool side effects so a destructive or crashing command cannot erase the record of the command request.

Required behavior:

  • Append before side effects.
  • Restore stored session state on resume instead of rebuilding it from new defaults.
  • Preserve the original prompt/profile snapshot for a resumed session unless the user explicitly migrates it.
  • Clean dangling tool-call tails before resume.
  • Keep enough metadata for audit and debugging.

[Edit pass: Added]. If your business requirements allow it, prefer files for persistance over databases. 

2.6 Verification and Feedback

Required in principle; stronger gates are minimum-useful for coding and high-impact workflows.

At minimum, every tool result must feed back into the next model call. The model should see actual exit codes, API responses, test output, and errors.
[Edit pass: I would word it as " every thing should go into the context", not "into the next model call"] 

For software work, a serious harness needs an evidence ledger:

  • Mark edits as making verification stale.
  • Record test/lint/build commands that passed after the last edit.
  • Block or bounce "done" claims when fresh verification is missing.
  • Keep verification evidence separate from model assertions.

Anthropic's long-running-agent guidance emphasizes explicit end-to-end testing because agents otherwise mark work complete too early. The supplied Hermes summary reports a verify-on-stop gate that checks evidence before accepting a final answer.

[Edit pass: Added]. The Evaluator should be external. Split the evaluator from the generator into distinct agents. 

2.7 Error Classifier

Required beyond toy use.

Do not treat all errors as "try again." Classify them into action hints:

  • Retry with backoff.
  • Compress context.
  • Rotate credential.
  • Fail over provider.
  • Ask user.
  • Abort as policy or quota failure.

The classifier belongs in the harness, not in every provider adapter or tool handler.

2.8 Observability and Accounting

Required for any multi-step or multi-user harness.

Record:

  • Model calls.
  • Tool calls.
  • Approval requests and decisions.
  • Token/cost usage.
  • Errors and retries.
  • Compactions.
  • Verification commands.
  • Final exit reason.

OWASP recommends logging agent decisions, tool calls, outcomes, token usage, costs per session/user, approval metadata, and unusual behavior. Without accounting, budgets are unenforceable.

3. Minimum-Useful Components

These are not required for a nine-line demo, but they become required quickly in real work.

3.1 Context Management

Use an escalation ladder:

  1. Truncate large tool outputs at ingest.
  2. Offload oversized outputs to sidecar files with previews.
  3. Drop or summarize old tool results.
  4. Compact the middle of the conversation.
  5. Reset into a fresh context using task/progress files.

OpenAI supports server-side and standalone compaction for long-running Responses workflows. Anthropic's context engineering guidance frames context as a finite attention budget and recommends using the smallest high-signal set of tokens.

3.2 Task and Plan State

[Edit pass: Added]. This is the outer loop mentioned above. 

Required for run-until-done autonomy.

A plan is not just UI. It is model-visible state that prevents the agent from attempting everything at once or declaring victory too early.

Minimum form:

  • A task checklist.
  • One in-progress item at a time.
  • Progress notes.
  • Verification evidence per item.
  • Resume instructions.

Anthropic's long-running-agent harness used an initializer agent to create feature requirements and a progress file, then asked later coding agents to work incrementally and leave structured updates

3.3 Interrupts and Steering

Minimum for interactive use.

The harness should let the user:

  • Interrupt a turn.
  • Deny or approve a pending action.
  • Add steering while a turn is running.
  • Pause and resume.

Steering should become input to the next model iteration, not a hidden out-of-band mutation.

3.4 Streaming

Minimum for good interactive UX; optional for headless batch runs.

Streaming adds complexity:

  • Partial tool-call accumulation.
  • Broken JSON repair.
  • Provider quirks.
  • Cancellation.
  • Stale stream detection.
  • Ordered result reconciliation.

Hermes is a good example of why streaming is a harness problem: provider adapters normalize chunks, repair tool calls, scrub hidden thinking tags, and convert stream state into a downstream response shape.

4. Optional Recommended Conditional Components

[Edit pass: 4.1 removed, it was specific to something I was doing] 

4.2 Memory

Optional for short tasks; useful for repeated work; risky if unbounded.

Useful memory types:

  • Working memory: current context window and active plan.
  • Conversation history: append-only transcripts.
  • Episodic memory: what happened in prior sessions.
  • Semantic memory: durable facts, preferences, project conventions.
  • Procedural memory: reusable workflows, often promoted into skills.
  • Task memory: goals, plans, progress files, evidence.
  • Environment memory: setup notes, repo-specific gotchas, command history.
  • Reflection memory: post-turn learnings.

Recommended memory API:

snapshot(profile, project) -> stable summary loaded at session start
recall(query, scope) -> selective memories injected late
write(candidate, kind, provenance) -> pending memory
consolidate(scope) -> dedupe, age, summarize, archive

OWASP warns about memory poisoning and recommends validation, isolation, expiration, size limits, and sensitive-data audits before persistence. OpenAI's Codex memory docs similarly treat memories as a helpful recall layer, not the sole source of required team rules.

4.3 Skills

Optional, but often the cheapest extensibility layer.

A skill is a reusable instruction bundle with optional references or scripts. Codex skills use progressive disclosure: the model sees a catalog first and reads the full skill only when selected. This is a better default than stuffing every workflow into the system prompt.

Use skills for:

  • Repeatable workflows.
  • Domain-specific procedures.
  • Artifact templates.
  • Tool usage playbooks.

Do not use skills for hard security controls. Enforcement belongs in policy, sandboxing, and hooks.

4.4 Plugins

Optional distribution layer.

Plugins package one or more of:

  • Skills.
  • App integrations.
  • MCP server config.
  • Assets.
  • Lifecycle hooks.

Use plugins when you need installation, sharing, marketplace metadata, or bundled integrations. Do not use plugins as the base abstraction inside the harness; they are packaging, not the control plane.

4.5 MCP and Connectors

Optional integration layer.

MCP is an open standard for connecting AI applications to external systems. Its architecture separates host, client, and server: the host coordinates LLM integration, permissions, context aggregation, consent, and multiple isolated client connections; servers expose resources, tools, and prompts with focused responsibilities.

Use MCP when:

  • External tools already speak MCP.
  • You need a common protocol for many data/tool providers.
  • You want remote or local servers to expose capabilities cleanly.

Keep the harness host in charge. MCP servers should not see the whole conversation or other servers. Large MCP catalogs should be deferred behind tool search or discovery rather than included in every prompt.

4.6 Hooks

Optional, but valuable once policy needs deterministic extension.

Hooks run at lifecycle points such as:

  • User prompt submit.
  • Pre-tool use.
  • Permission request.
  • Post-tool use.
  • Pre/post compaction.
  • Stop.
  • Subagent start/stop.

Use hooks for logging, policy checks, data-loss prevention, validation, custom prompt injection, or stop gates. Hooks should be reviewed and trusted because they execute host-side code.

4.7 Subagents

Optional for simple tasks; useful for parallel read-heavy exploration and context isolation.

A subagent is a separate harness run with:

  • Its own context.
  • Its own budget.
  • A restricted tool set.
  • A bounded task.
  • A distilled result returned to the parent.

Use subagents when:

  • Work can be split safely.
  • Intermediate output would pollute the main context.
  • Different specialists need different tools or prompts.
  • You need parallel review or triage.

Avoid parallel write-heavy subagents unless the harness has strong merge coordination.

4.8 Parallel Tool Execution

Optional.

Start sequential. Add parallelism only after tools declare mutability:

  • Read-only tools can run concurrently.
  • Mutating tools should run sequentially unless path/resource scopes do not overlap.
  • Results should be appended in model-emission order, not completion order.

4.9 Guardian or Reviewer Models

Optional.

A reviewer model can classify approval requests, inspect tool arguments, or evaluate final claims. It should never replace sandboxing. It is a reviewer at the boundary, not the boundary itself.

4.10 Checkpoints and Rollback

Optional for read-only work; useful for autonomous file edits and risky operations.

Checkpoint before:

  • Destructive shell commands.
  • Large edits.
  • Database writes.
  • External side effects.

The checkpoint system should be outside model control.

6. Canonical Call Order

6.1 Session Start

Call order:

  1. Resolve frontend request: CLI, app, IDE, API, scheduler, chat connector.
  2. Resolve profile or automation identity.
  3. Resolve workspace/project.
  4. Load enforced organization/admin policy.
  5. Load profile policy, budgets, model preferences.
  6. Load project instructions.
  7. Load profile persona and user model.
  8. Load memory snapshot if enabled.
  9. Discover tools, skills, MCP servers, and plugins.
  10. Apply visibility, allow/deny, and availability probes.
  11. Open or create session history.
  12. Freeze session prompt prefix and cache key namespace.

Do not call credentials yet. Credentials are per action, not per session.

6.2 Turn Prologue

Call order:

  1. Receive user message or scheduled task.
  2. Run UserPromptSubmit hooks, if configured.
  3. Append user message to canonical history.
  4. Reset per-turn counters.
  5. Mint per-turn budget.
  6. Refresh tool/MCP availability if needed.
  7. Rehydrate task state and progress files.
  8. Run preflight context-size check.
  9. Compact or summarize if over threshold.
  10. Build request context.

6.3 Request Build

Call order:

  1. Start with stable system/developer instructions.
  2. Add harness operating rules.
  3. Add permission/sandbox description the model should know.
  4. Add environment context: cwd, date, writable roots, platform.
  5. Add project instructions in precedence order.
  6. Add profile snapshot: persona, user model, memory summary.
  7. Add skill index and selected skill instructions.
  8. Add task/plan state.
  9. Add conversation history or conversation ID.
  10. Add late recalled memory and dynamic context.
  11. Add current user message.
  12. Add tool schemas for visible tools.
  13. Add cache metadata.
  14. Send model request.

If using OpenAI Responses directly, the application owns this loop. If using the Agents SDK, the SDK owns more of the loop and exposes agents, sessions, guardrails, handoffs, approvals, and traces.

6.4 Model Stream Handling

Call order:

  1. Receive deltas.
  2. Accumulate text.
  3. Accumulate tool-call names and arguments.
  4. Normalize provider-specific stream state.
  5. Emit user-visible partial text only when safe for the UI.
  6. On completed response item, classify it as message or tool call.
  7. If no tool calls remain, move to stop gates.
  8. If tool calls exist, dispatch each through the tool funnel.

6.5 Tool Call Funnel

Call order:

  1. Parse tool name and arguments.
  2. Resolve deferred tool, if using tool search.
  3. Validate schema.
  4. Record requested call to history.
  5. Run PreToolUse hooks.
  6. Classify action risk.
  7. Ask policy provider: allow, ask, or deny.
  8. If ask, route to user or reviewer agent.
  9. If approved, bind approval to exact normalized action.
  10. Fetch scoped credentials if needed.
  11. Start sandboxed or external execution.
  12. Enforce timeout and cancellation.
  13. Capture stdout/stderr/API result.
  14. Redact secrets.
  15. Truncate or offload large output.
  16. Fence untrusted external data.
  17. Run PostToolUse hooks.
  18. Append result to history.
  19. Return result into the next model request.

Credentials are intentionally late. Policy and approval decide whether the action is allowed before any secret is fetched.

6.6 Loop Continuation

Call order:

  1. Check interruption.
  2. Check budget.
  3. Check guardrails for repeated failures or no progress.
  4. Check context size.
  5. Compact if needed.
  6. Build next request with tool results included.
  7. Call model again.

6.7 Stop Gates and Finalizer

Call order:

  1. Model produces no tool calls.
  2. Run empty-response or "I will do it" recovery checks, if configured.
  3. Run verification gate if edits or high-impact actions occurred.
  4. Run Stop hooks.
  5. If a stop gate asks for continuation, append a synthetic user/tool message and continue the loop.
  6. If accepted, finalize answer.
  7. Persist final turn state.
  8. Record usage and cost.
  9. Update task state.
  10. Trigger post-turn memory/skill reflection in a restricted background worker, if enabled.
  11. Return final response to frontend.

Memory writeback should happen after the turn, not mid-request, unless the user explicitly asks to remember something. Even then, save a pending memory candidate and validate it before consolidation.

6.8 Subagent Delegation

Call order:

  1. Parent decides delegation is useful.
  2. Parent creates subtask prompt with explicit success criteria.
  3. Harness creates child profile view: same user ceiling, narrower tools, separate budget.
  4. Child session starts with no access to full parent transcript unless explicitly summarized.
  5. Child runs normal loop.
  6. Child approval requests route to parent/user policy.
  7. Child returns distilled result and evidence.
  8. Parent appends summary, not raw logs.
  9. Parent decides next action.

The parent should not hand children broad credentials, broad tools, or unrelated memory.

8. Codex and Hermes as Reference Shapes

8.1 Codex Pattern

Codex, as reflected in the supplied local trace and current docs, is a strong example of a security-first coding harness:

  • Frontends submit operations to a session; events stream back.
  • Stable base instructions stay outside mutable conversation history.
  • Environment and permission context are explicitly described.
  • Tools are computed per request.
  • Tool calls route through typed dispatch.
  • Approval and sandbox policy are separate.
  • Tool results feed the next Responses request.
  • Rollout persistence enables resume.
  • Subagents use the same queue/event primitive with parent-routed approvals.

Best transferable idea: keep the safety gate as a deterministic pipeline: parse, record, hook, classify, approve, sandbox, execute, normalize, record.

8.2 Hermes Pattern

Hermes, as reflected in the supplied local trace, is a strong example of a multi-provider, multi-frontend harness:

  • Many frontends enter one engine method.
  • Prompt tiers are cache-stable.
  • Dynamic memory/plugin/advisor content is injected late.
  • Provider transports normalize model differences.
  • Tool dispatch has concurrency analysis and output budgeting.
  • Guardrails detect repeated failures.
  • Verify-on-stop prevents unsupported completion claims.
  • Background self-improvement extracts memories and skills after the turn.

Best transferable idea: treat provider behavior, verification, and memory learning as harness concerns, not one-off model prompt instructions.

8.3 Where They Agree

Both agree on the critical architecture:

  • One central loop.
  • Stable prompt prefix.
  • Ordered context assembly.
  • Tool schemas as prompt surface.
  • One dispatch funnel.
  • Safety before execution.
  • Tool outputs as ground truth.
  • Append-only history.
  • Context management.
  • Optional subagents for bounded delegation.

9. Build Order

v0: Minimal Acting Harness

Build:

  1. Model adapter.
  2. Agent loop with max iterations.
  3. Static system prompt.
  4. Four tools: shell, read, write/patch, search/list.
  5. Sequential dispatch funnel.
  6. Basic sandbox.
  7. JSONL history.
  8. Error-as-result behavior.
  9. Final answer exit reason.

Do not add plugins, subagents, vector memory, or parallelism yet.

v1: Daily-Use Harness

Add:

  1. Prompt-cache discipline.
  2. Tool output truncation/offload.
  3. Context compaction.
  4. Approval policy.
  5. Verification ledger.
  6. Task/plan state.
  7. Resume.
  8. Token/cost accounting.
  9. Error classifier with retry/compress/abort hints.
  10. Interrupts.

This is the level where the harness becomes reliable for real work.

v2: Extensible Harness

Add:

  1. Memory provider.
  2. Skill library.
  3. MCP client support.
  4. Hooks.
  5. Streaming.
  6. Multi-provider routing.
  7. Tool availability probes.
  8. Parallel read-only execution.

v3: Autonomous or Multi-User Harness

Add:

  1. Per-user budgets.
  2. Per-user telemetry.
  3. Short-lived credential provider.
  4. Subagents.
  5. Checkpoints and rollback.
  6. Guardian/reviewer models.
  7. Memory consolidation.
  8. Evals and replayable traces.
  9. Admin policy and managed configuration.

10. Practical Design Recommendations

  1. Own the loop until you know why you do not want to. Frameworks are useful, but the harness's correctness depends on details hidden by many frameworks: prompt contents, tool outputs, approval state, and retry behavior.

  2. Keep the model prompt descriptive, not authoritative. The prompt can explain policy to the model, but the harness must enforce policy.

  3. Make authority late-bound. Credentials, approvals, and write permissions should be resolved only at the action boundary.

  4. Make context stable by default. Stable prefix first, volatile content late, dynamic retrieval on demand.

  5. Prefer files before databases for a light harness. Markdown/JSONL task files, memory files, and session logs are inspectable and easy to debug. Add indexes later.

  6. Treat memory as untrusted input. Validate, scope, expire, and audit. Memory poisoning is persistent prompt injection.

  7. Use subagents to protect context, not to create organizational theater. Subagents are useful when they isolate noisy work and return concise evidence.

  8. Make verification evidence mechanical. A test pass, exit code, browser trace, or API result is evidence. A model claim is not.

  9. Prefer small tool sets with clear contracts. Anthropic's tool guidance and the local Hermes trace both show tool descriptions are prompt engineering. Bloated overlapping tools confuse models.

  10. Keep logs and caches inside the trust boundary. Prompt-cache keys, traces, and telemetry can contain or reveal profile data.
