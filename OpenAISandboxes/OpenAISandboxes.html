ChatGPT Codex vs. ChatGPT Work:
Sandbox Architectures Compared

What each mode of the unified ChatGPT desktop app allows by default on macOS and Windows, how to expand or override the boundaries, and how code execution, web tools, and connector (MCP) calls behave differently in each.

Compiled July 2026 · Sources: OpenAI docs & Help Center, engineering posts, third-party security research · Companion to "Claude Code vs. Claude Cowork"

A note on naming On July 9, 2026, OpenAI merged the standalone Codex app into a new unified ChatGPT desktop app for macOS and Windows, with three modes: Chat, Work, and Codex. The previous ChatGPT desktop app was renamed ChatGPT Classic.[6][11] "ChatGPT Codex" below refers to the Codex mode (and its CLI/IDE siblings, which share the same sandbox stack); "ChatGPT Work" refers to the new agentic Work mode. Most of the sandbox architecture predates the merger under the legacy names Codex and ChatGPT agent, and OpenAI's docs still describe it that way.

01 · Thesis

The core architectural difference

The two modes now share one desktop shell, one account, and one plugin/connector surface, but they retain distinct execution architectures.[10] Codex is local-first: it works with local folders, repositories, terminals, and developer tools, is available only as a desktop mode (not on web or mobile), and runs model-generated commands inside an OS-enforced sandbox on your machine.[5][1] Work is cloud-first: it runs code and shell commands in a managed, isolated environment on OpenAI's side, is available on web and mobile as pure cloud execution, and uses the desktop app as a permissioned bridge when a task needs local files or desktop apps.[2][5]

Codex's security model is explicitly two-layer: a sandbox mode defines what commands can technically do (where they can write, whether they can reach the network), and an approval policy defines when the agent must stop and ask before crossing a boundary.[1] The sandbox applies to spawned commands, not just built-in file operations — a notable difference from Claude Code, whose built-in sandbox covers only the Bash tool.[2] Codex is also the only major CLI coding agent that ships with sandboxing on by default.[9]

02 · Product deep dive

ChatGPT CodexDefaults and platform behavior

Default state: sandboxed, offline, workspace-scoped

By default the agent runs with network access turned off, inside an OS-enforced sandbox that limits writes to the current workspace, plus an approval policy controlling when it must ask before acting.[1] The default Auto preset (--sandbox workspace-write --ask-for-approval on-request) lets Codex read files, make edits, and run commands in the working directory automatically, asking for approval to edit files outside the workspace or run commands that need network access.[1] The workspace includes the current directory plus temp directories like /tmp; even in workspace-write mode, certain paths stay read-only — .git internals (recursively, including hooks), .agents/, and .codex/ — so a sandboxed command can't tamper with its own policy or repository plumbing.[4] The other presets are read-only and danger-full-access.[1]

Platform support — including native Windows

Codex uses platform-native enforcement on each OS.[2] macOS uses Seatbelt policies, running commands via sandbox-exec with a profile matching the selected sandbox mode. Linux uses bwrap plus seccomp by default (the sandbox moved from Landlock to bwrap in v0.115). On Windows, WSL2 gets the Linux implementation — and, unlike Claude Code, native Windows has its own sandbox.[1] OpenAI's May 2026 engineering post describes it: an elevated mode (preferred) using dedicated lower-privilege sandbox users, ACL-based filesystem boundaries, and Windows Firewall rules for network control after a one-time admin setup; and an unelevated fallback using a restricted token derived from the current user, with weaker network isolation — unsupported policies are refused rather than silently weakened.[3][8]

Expanding or overriding the boundaries

  • Permission profiles. config.toml profiles define filesystem rules (read / write / deny, with glob support and deny > write > read precedence) and network domain allow/deny lists — e.g. keep the workspace writable while denying all **/*.env files. Profiles can extend :read-only or :workspace but can never extend :danger-full-access.[3]
  • Widening scope. --add-dir or writable_roots add writable paths; [sandbox_workspace_write].network_access = true enables network for sandboxed commands. Local and private network targets stay blocked by default unless explicitly allowlisted.[3][4]
  • The escalation flow. If a sandboxed command fails on a permissions error, Codex asks whether to retry unsandboxed; approving marks the command trusted and re-executes it with no sandbox.[12] Approval cadence is tunable (on-request, unless-trusted, on-failure, never).[1]
  • Skipping everything. --sandbox danger-full-access and --dangerously-bypass-approvals-and-sandbox (aka --yolo) remove the boundaries; OpenAI's docs frame these for containers and other externally isolated environments.[4]

Projects, folders, and the cloud variant

Like Claude Code, a Codex "project" is fundamentally the trusted working directory (Codex may start read-only until you explicitly trust the folder), extended by profile-defined workspace_roots for multi-repo work.[4][3] Codex cloud is a different animal: tasks run in isolated OpenAI-managed containers with a two-phase runtime — a setup phase with network access to install dependencies, then an agent phase that runs offline by default unless internet access is enabled for that environment (optionally with a domain allowlist). Secrets configured for cloud environments are available only during setup and are removed before the agent phase starts.[1]

03 · Product deep dive

ChatGPT WorkDefaults and platform behavior

Architecture

Work runs code and shell commands in a managed, isolated environment on OpenAI's infrastructure, with workspace policy and tool-specific controls determining which capabilities are available.[2] On web and mobile, Work runs entirely in the cloud and cannot directly access files on your computer. In the desktop app, Work can additionally use local files and desktop apps with your permission — and, notably, desktop Work threads and local files remain on that computer, while cloud Work conversations stay in the cloud (at launch the two don't sync).[5][6]

Default file access: grants, not mounts

Like Cowork, Work does not begin with an automatic mount of your filesystem. On desktop you open a local folder or project and grant access only to the files the task needs; local files and outputs remain on that computer unless you explicitly move or share them.[5] Work also ships a built-in browser for gathering information from websites and working with web-based tools — this browser does not inherit your Chrome profile, cookies, or signed-in sessions, and ChatGPT asks before touching a new website unless you've already allowed that host or chosen a more permissive setting.[6][13]

Network defaults

For code and shell commands, network access is governed by Settings → Data controls → Work network access, where "Allow public internet access" opens the execution environment to the public internet; changes take effect after the current run finishes and the environment refreshes. Web search, apps and connectors, and the remote browser have separate controls — the code-execution network toggle does not govern them.[2] Enterprise administrators can restrict browser access, plugin access, local file access, and agent network permissions at the workspace, group, and user level, and OpenAI has added an Auto-Review layer — a second model that checks important actions involving connected tools before they execute.[10]

Computer use

Computer Use is available in the desktop app in both Work and Codex modes on macOS and Windows, installed as a plugin. It operates graphical interfaces on the host — requiring Screen Recording and Accessibility permissions on macOS — with per-app permission prompts, an "Always allow" list (persisted in config.toml on Windows), and hard exclusions: it can't automate terminals or ChatGPT itself, and can't authenticate as an administrator or approve security prompts.[7] As with Cowork, this capability necessarily operates outside the isolated execution environment, because its whole purpose is to act on your real desktop.[7]

04 · Runtime behavior

How code, web search, and MCP behave differently

Three asymmetries matter most in practice — and they rhyme closely with the Claude Code / Cowork pair.

1 · What the sandbox actually covers

Codex's local sandbox governs sandboxed shell commands. Connectors, MCP servers, browser and computer-use surfaces, cloud environment settings, and approved escalations all use their own separate controls — the permission profile does not constrain them.[3] Work inverts the picture: code and shell are structurally contained server-side, but the desktop bridge (local files, desktop apps, computer use) is where the host-side risk lives. In both modes, computer use is the uncontained surface, acting directly on your real desktop.[7]

2 · Web tools have their own rules

Commonly missed In Work, the code-execution network toggle does not govern web search, connectors, or the remote browser — each has separate controls.[2] In Codex, web search defaults to cached mode: an OpenAI-maintained index returns pre-indexed results instead of fetching live pages, reducing exposure to prompt injection from arbitrary live content. Live browsing requires --search or web_search = "live" — and if you're running full-access, web search silently defaults to live.[4]

3 · MCP / connector calls

Since the merger, Chat, Work, and Codex share the same plugin and connector surface — the same Slack, Google Drive, GitHub, and Salesforce connections now reach all three modes, which is what makes the merger structurally significant for the trust boundary.[10] Codex elicits approval for connector tool calls that advertise side effects, and destructive MCP tool calls always require approval when the tool advertises a destructive annotation — even if it also advertises read-only hints.[1] In Work, connector actions are additionally screened by the Auto-Review model for important actions, plus admin-configurable restrictions per workspace.[10]

One notable contrast with the Anthropic pair: on native Windows, Codex ships a real OS sandbox (restricted tokens, ACLs, firewall rules) where Claude Code offers none without WSL2 — while Cowork counters with a full Hyper-V VM that Codex's token-based approach doesn't match in isolation strength.

06 · Limits

The honest caveats

The merger itself is the biggest one. Codex's original design was deliberately bounded — cloud tasks cloned a repository into an isolated environment, ran offline by default, and never touched the host — and security analysts note that the unified app now places code execution, browser access, local files, and enterprise connectors inside a single agent runtime, a significantly larger trust boundary than the original sandboxed Codex environment.[10] OpenAI itself has acknowledged that prompt injection is unlikely to ever be fully solved, positioning the Auto-Review layer as mitigation rather than elimination.[10]

Two patched vulnerabilities from early 2026 illustrate the boundary-testing that's already occurred: a command-injection flaw in GitHub branch-name handling, and a DNS-based side-channel exfiltration path in the code-execution environment.[10] And a freshness caveat: ChatGPT Work is days old at time of writing — its documentation is still being consolidated under the new naming, rollout is staged across plans, and specifics (especially cloud/desktop sync behavior and enterprise controls) are explicitly marked "at launch" in OpenAI's own Help Center. Verify against current docs before relying on any single detail.[5]
