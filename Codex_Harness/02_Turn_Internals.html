00 The shape · four nested loops

run_turn owns context and continuation; run_sampling_request owns one HTTP/WS request; the innermost loop consumes the model's token & tool-call stream. Understanding which loop owns what makes the rest fall into place.

01 run_turn · phase by phase

Everything that shapes context happens here — before and around the model call.

A

Pre-sampling compaction run_pre_sampling_compact · turn.rs:797

Runs first, and can fire the model before your prompt is even sent. Two triggers:

  • Previous-model inline compact :835 — if you switched models and either the compaction-compatibility hash changed (comp_hash_changed :825) or you downshifted to a smaller context window, history is compacted against the previous model first.
  • Token-limit check :807 — if the auto-compact budget or usable window is already exhausted → run_auto_compact(… ContextLimit, PreTurn) at :917. Compaction is itself a model call that summarizes history into a Compaction item.
B

Context capture & injection record_context_updates… · mod.rs:3549

This decides how much context to inject, and it's where memories and the environment block land.

  • First turn of a context window → should_inject_full_context = true → build_initial_context_with_world_state_and_mcp mod.rs:3145 assembles the full bundle and sets it as the reference_context_item baseline.
  • Every later turn → only diffs: settings updates (model / approval / sandbox changes) + a world-state diff. This is why the big environment block appears once, not every turn.

The initial bundle fills three lists — developer_sections, contextual_user_sections, separate_developer_sections — in order mod.rs:3145–3406:

  • PermissionsInstructions :3180 — renders the current approval policy + sandbox + execpolicy so the model knows what it may do.
  • Developer instructions :3206 — where AGENTS.md-derived user instructions and any developer prompt land.
  • Collaboration mode, realtime, Personality, Apps, Skills catalog :3255, plugins.
  • Context-contributors loop :3314 — contribute_thread_context per extension. This is the memory injection point.
  • World state :3400 — world_state.render_full() emits the environment context (cwd, sandbox mode, workspace roots, git info).

Finally :3411 developer_sections is joined into one developer message and contextual_user_sections into a user-role message, then appended to history.

How memories get in

Memories are an extension (ext/memories), not hardcoded. MemoriesExtension::contribute_thread_context (extension.rs:48) calls build_memory_tool_developer_instructions (ext/memories/src/prompts.rs:26), which reads <codex_home>/memories/memory_summary.md from disk, truncates it to a token budget, and renders the read_path.md template. Only the summary is injected up front, once per context window — full memory files are pulled later when the model calls the memory read/search tools. A cited memory emits an <oai-mem-citation> block that strip_hidden_assistant_markup_and_parse_memory_citation (stream_events_utils.rs:79) strips from visible text and parses.

C

Turn-scoped injections build_skills_and_plugins · turn.rs:510

Unlike the once-per-window base context, this runs every turn, driven by this turn's user input.

  • Scans the message for explicit @skill and plugin:// mentions and injects only the mentioned skills'/plugins' instructions.
  • maybe_prompt_and_install_mcp_dependencies :591 — a mentioned skill can require an MCP server; this may prompt to install it.
  • build_extension_turn_input_items :682 — runs every turn-input contributor. Memories is a context contributor, not a turn-input one, so it is not re-run here.
  • Guardian sessions short-circuit this :519 so untrusted transcript text can't smuggle mentions.
D

Hooks & input recording run_hooks_and_record_inputs · turn.rs:481

After run_pending_session_start_hooks, each input item runs through inspect_pending_input (user-configured hooks). A hook can block the input (recording an "additional context" note instead) or let it through to be recorded as pending input. A turn with only blocked input and no accepted user text ends early.

E

The loop turn.rs:225

Each iteration:

  • Drain queued / steer input into history (get_pending_input) — how a message typed while the model runs gets folded in.
  • Emit reminders: rollout_budget::maybe_record_reminder, maybe_record_current_time_reminder.
  • capture_step_context, build input via clone_history().for_prompt(…), then run_sampling_request.
  • On return: mid-turn auto-compact if needs_follow_up && (new_context_requested || token_limit_reached) → run_auto_compact(… MidTurn) then continue :349.
  • Stop hooks :373 can inject a continuation prompt and force another iteration.
  • needs_follow_up :318 = model didn't end its turn OR pending input exists. False → break, return last_agent_message.

Notable defense: an InvalidImageRequest :421 rewrites the last turn's images to a placeholder and retries — guarding against image-based prompt poisoning.

02 run_sampling_request · one model call

Assemble tools, build the prompt, stream the response, run the tools.

1

Build the tool set built_tools · turn.rs:1177

Lists all MCP tools, resolves connectors/apps, computes tool-suggest candidates, then build_mcp_tool_exposure splits MCP tools into direct (shown to the model) vs deferred (reachable only via tool-search) — the mechanism that keeps a huge MCP inventory from flooding context. Returns a ToolRouter :1299.

2

Build the prompt build_prompt · turn.rs:1044

The Prompt is assembled from four things and nothing else:

Prompt {
    input:            history.for_prompt(input_modalities),   // the conversation
    tools:            router.model_visible_specs(),           // only "direct" tools
    parallel_tool_calls: model_info.supports_parallel_tool_calls,
    base_instructions,                                         // the system prompt
    output_schema:    turn_context.final_output_json_schema,   // optional structured-output
}

base_instructions (get_base_instructions · mod.rs:1263) is simply session_configuration.base_instructions — the system-prompt text, chosen per model / personality.

3

Is the prompt "rewritten"? for_prompt → normalize_history · history.rs:141 / 359

There is no semantic rewriting of your messages. The only transformation when turning history into model input is normalization :359:

  • ensure_call_outputs_present — every tool call must have a matching output;
  • remove_orphan_outputs — every output must have a matching call;
  • strip_images_when_unsupported — drop images if the model can't take them.
The nuance

Tool outputs are separately truncated to a token budget when recorded (process_item / truncate_function_output_payload · :370). So "rewriting" means structural repair + truncation + image stripping, never paraphrasing. The real content shaping happens earlier — via context injection (Phase B) and compaction (summarization).

4

Retry loop & streaming try_run_sampling_request · turn.rs:1887, loop :1945

The retry loop :1103 re-runs on retryable stream errors with backoff (up to stream_max_retries); ContextWindowExceeded and UsageLimitReached return immediately. The inner loop opens client_session.stream(prompt, …) and consumes ResponseEvents:

  • OutputItemAdded/Done → handle_output_item_done. Tool calls become futures pushed onto in_flight :2067 — a FuturesOrdered, so multiple calls run concurrently but resolve in order.
  • OutputTextDelta → streamed to the UI as AgentMessageContentDelta.
  • SafetyBuffering / ServerModel / ModelVerifications / TurnModerationMetadata → surfaced as events (server-side moderation / routing signals).
  • Completed :2212 → records token usage; needs_follow_up = true iff end_turn == Some(false) — the model signaled it isn't done.

The in_flight futures are awaited by drain_in_flight turn.rs:1853: each result becomes a ResponseInputItem recorded into history — exactly what the next sampling request picks up via for_prompt. That's the tool-result-back-to-model handoff.

03 Where & when security is checked

The one thing to remember

Security is not checked at prompt-build time. It is checked at tool-execution time, per call, inside the ToolOrchestrator — whose own doc comment calls it "approvals + sandbox selection + retry."

The path from a streamed tool call to enforcement: handle_output_item_done · stream_events_utils.rs:405 → ToolRouter::build_tool_call :413 → handle_tool_call :436 → router dispatch → ToolOrchestrator::run · orchestrator.rs:134. The orchestrator's sequence:

1

Approval requirement orchestrator.rs:156

Each tool declares an ExecApprovalRequirement, defaulting per AskForApproval. Skip → auto-approved (unless strict-auto-review forces a Guardian pass). Forbidden → ToolError::Rejected at once. NeedsApproval → if routes_approval_to_guardian or strict-auto-review is set :149, the Guardian LLM auto-approver runs first; otherwise the user is prompted. reject_if_not_approved :217 aborts on denial.

2

Sandbox selection orchestrator.rs:224

should_sandbox + select_initial pick a SandboxType — Seatbelt (mac) / Landlock+seccomp (linux) / WindowsRestrictedToken / None — from the file-system & network policies, the tool's sandbox_preference, and workspace roots. The SandboxAttempt :260 carries the permission profile, writable roots, and sandboxed cwd.

3

First attempt run_attempt · orchestrator.rs:280

The command runs under the selected sandbox. Success → return output straight back to the model.

4

Denial escalation orchestrator.rs:297

On SandboxErr::Denied: a network denial maps to a NetworkApprovalContext (host to approve). If the tool doesn't escalate_on_failure, or the policy forbids unsandboxed retry (Never/OnRequest without a network-prompt exception :339), the denial returns as-is — the model sees the error. Otherwise it builds a retry reason and, unless a bypass applies :389, requests approval again (strict-auto-review needs a fresh Guardian review, since the first only covered the sandboxed attempt), then retries with an escalated sandbox — often SandboxType::None :419.

The two tool families check safety differently

exec / shell

The execpolicy engine (exec_policy.rs) renders Allow / Prompt / Forbidden, which feeds the orchestrator's approval requirement — then the full sandbox gauntlet above.

apply_patch

assess_patch_safety (safety.rs:32) → AutoApprove{sandbox_type} / AskUser / Reject. Core rule :66: if the patch is constrained to writable paths AND a platform sandbox exists, auto-approve inside a sandbox (guards against hard-link escapes). No enforceable sandbox → ask, or reject under Never. UnlessTrusted always asks.

The AskForApproval postures

NeverPrompts become rejections; no unsandboxed retry.
UnlessTrustedOnly known-safe / read-only actions auto-run; everything else asks.
OnRequest ◂ defaultAuto-run in sandbox where possible; escalate on denial (Guardian may auto-resolve).
Granular(cfg)Per-category toggles: sandbox_approval, rules, skill_approval, request_permissions, mcp_elicitations.
