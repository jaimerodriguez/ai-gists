Claude Code vs. Claude Cowork:
Sandbox Architectures Compared

What each product allows by default on macOS and Windows, how to expand or override the boundaries, and how code execution, web tools, and MCP servers behave differently in each.

01 · Thesis

The core architectural difference

Claude Code and Claude Cowork sit at opposite ends of the isolation spectrum. Claude Code runs on the user's machine with filesystem, shell, and network access, contained via OS-level sandboxing (Seatbelt on macOS, bubblewrap on Linux) plus trust dialogs where reads are allowed but writes and bash require approval.[1] Claude Cowork, by contrast, runs a full VM — Apple's Virtualization framework on macOS, HCS (Host Compute Service, the layer underneath Hyper-V) on Windows.[1][9]

The design philosophy differs because the audiences differ. Claude Code targets developers who can read bash, so it uses a human-in-the-loop OS sandbox that allows reads, confines writes to the workspace, and filters network through a proxy. Cowork targets knowledge workers and puts everything inside a sealed VM. Anthropic's engineering retrospective notes the key structural distinction: in Cowork's original full-VM mode, Claude executed as an ordinary Linux user with no awareness it was sandboxed, whereas in Claude Code a privileged process sits outside the sandbox deciding per-command whether to enforce it — meaning a persuasive injected prompt or a fatigued approval click can get that process to run something un-sandboxed. In the VM, no outer process holds an escape-hatch key.[1]

02 · Product deep dive

Claude CodeDefaults and platform behavior

Default state: sandbox off

If you've never run /sandbox, every bash command Claude runs has the same access your terminal does — the sandbox is off until you turn it on.[10] The baseline protection is the permission model: by default Claude Code is read-only and asks for permission before making modifications or running commands, auto-allowing only safe commands like echo or cat.[2]

With the sandbox enabled (macOS / Linux / WSL2)

Defaults become: read access to the entire computer except certain denied directories — note this default still allows reading credential files like ~/.aws/credentials and ~/.ssh/ unless you block them via sandbox.credentials or denyRead. Writes are blocked outside the current working directory and session temp directory, including shell config files and system binaries.[3] Network-wise, internet access is only allowed through a unix domain socket connected to a proxy server running outside the sandbox, filtered by a domain allowlist.[2] Every child process spawned by a sandboxed command inherits the same restrictions.[2]

Platform support

On macOS there is nothing to install — sandboxing uses the built-in Seatbelt framework. On Windows, you run Claude Code inside a WSL2 distribution; WSL1 and native Windows are not supported.[3] On native Windows, Claude Code today relies entirely on the permission-prompt layer, or on external isolation such as Docker containers or VMs.[4]

Expanding or overriding the boundaries

  • Expanding paths. Grant write access to additional paths using sandbox.filesystem.allowWrite in settings; custom allowed and denied paths (denyRead, allowRead) are fully configurable at user, project, or managed scope.[3]
  • The escape hatch. When a sandboxed command fails due to restrictions, Claude Code may retry with the dangerouslyDisableSandbox parameter, which prompts the user. Setting allowUnsandboxedCommands: false disables this entirely. Commands listed in excludedCommands always run outside the sandbox — used for tools like Docker that can't be sandboxed.[3]
  • Skipping everything. The --dangerously-skip-permissions flag bypasses all permission checks and should only be used in isolated environments like containers without internet access.[4]
  • Self-protection. The sandbox automatically denies write access to Claude Code's settings.json files at every scope, so a sandboxed command cannot modify its own policy.[3]

Projects and folders

"Projects" in Claude Code are simply the working directory: whatever directory you launch claude in becomes the writable workspace. You widen or narrow it via settings.json using sandbox.filesystem paths and permission allow/deny rules — paths from both sources are merged into the final sandbox configuration.[3]

03 · Product deep dive

Claude CoworkDefaults and platform behavior

Architecture

On macOS, Cowork runs inside a Linux virtual machine, isolated from the host OS, where the agent can execute code, process files, and run tools without direct access to your system.[8] On Windows, Cowork wraps Claude Code in a Hyper-V-isolated Ubuntu VM with several layers of protection: Authenticode-gated named-pipe RPC, bubblewrap namespaces, per-session unprivileged users, a seccomp filter, and a domain-restricted egress proxy.[9]

Cowork also supports remote sessions (beta): tasks run in an isolated, temporary environment on Anthropic's servers that can't reach your home or company network and is removed when the session ends. When a task needs a local file, Claude reaches your computer through the Claude Desktop app, only for folders you've connected, and only while the app is open.[5][7]

Default file access: nothing

This is the sharpest contrast with Claude Code's "read everything" default. Claude can only read and write files in folders you've explicitly connected — nothing is accessible by default — and network access follows the egress settings you've configured.[6]

Granting folder access

You connect folders through the Cowork interface: Claude prompts when it needs a folder and you approve each one. On macOS the OS adds its own layer — you may also need to verify Claude Desktop's permissions under System Settings → Privacy & Security → Files and Folders. Anthropic's own guidance is to create a dedicated working folder for Claude rather than granting broad access, and to keep backups of important files.[5]

Permission behavior is mode-dependent. Auto mode won't approve certain sensitive actions — connecting additional folders, deleting files, creating scheduled tasks — while Skip all approvals removes all pausing and checking. Regardless of mode, Claude always asks before permanently deleting files.[6][5]

Network defaults

All VM traffic is forced through controlled proxies. HTTPS traffic from the VM routes through a MITM proxy on the host using a fresh ephemeral CA generated each boot, and an outer host-side egress proxy filters outbound requests against a domain allowlist.[8] For remote sessions, the sandbox can't reach private, internal, link-local, or cloud-metadata addresses; all traffic passes through a mandatory proxy the sandbox can't reconfigure or bypass; and "no network access" is the default for Enterprise organizations.[7] Users expand this via the internet/egress allowlist settings, and Team/Enterprise admins can configure the egress allowlist centrally.[11]

04 · Runtime behavior

How code, web search, and MCP behave differently

Three asymmetries matter most in practice.

1 · What the sandbox actually covers

In Claude Code, the built-in sandboxed Bash tool restricts only Bash commands — the built-in file tools (Read, Edit, Write), MCP servers, and hooks still run directly on your host through the permission system, not the sandbox.[3][4] In Cowork, the entire code-execution environment lives inside the VM, so shell commands and scripts are structurally contained. But Cowork has its own exception: computer use has no sandbox between Claude and your screen — it runs on the host, not in the VM, unlike file operations (permission-checked) and code execution (VM-isolated).[5][8]

2 · Web search and fetch bypass egress rules in Cowork

Commonly missed Network egress permissions don't apply to the web fetch or web search tools or MCPs, including Claude in Chrome. Web fetch runs server-side and is limited to search results and URLs you've shared.[5] Your egress allowlist governs code the agent runs — not the built-in web tools or connectors.

In Claude Code, WebFetch and WebSearch are governed by permission rules like any other tool, and the sandbox's network rules apply to what bash-level commands like curl can reach.[3]

3 · MCP servers

In Claude Code locally, MCP servers execute on your host outside the bash sandbox, governed by permission rules rather than OS isolation. The only way to contain them is to put the whole Claude Code process inside a container or VM — every isolation approach other than the built-in Bash sandbox puts the entire process inside the boundary, so file tools, MCP servers, and hooks are restricted too.[4]

Cowork follows a different pattern. For remote sessions, connector authorization tokens never enter the sandbox; connector calls are made server-side, and the sandbox holds only session-scoped tokens that expire within hours.[7] On desktop, local connectors and plugins that include local MCP servers work through the desktop app only — meaning local MCP servers run on the host, outside the VM boundary, which is why Anthropic's safety guidance stresses caution with unfamiliar MCPs and plugins.[12][5]

06 · Limits

The honest caveat

The boundaries aren't absolute, and Anthropic's own retrospective is candid about it. In one disclosed incident, a malicious file exfiltrated data through an approved domain: allowing api.anthropic.com meant allowing file uploads to arbitrary Anthropic accounts. The fix was a defensive MITM proxy inside the VM that only passes requests carrying the VM's own provisioned session token, rejecting attacker-embedded keys.[1] The framing worth keeping: an allowlist is a capability grant, not a destination filter — every function reachable through an allowed domain becomes attack surface.[1]

Separately, VM isolation creates a monitoring gap: host EDR tools can't see inside the guest, and remote sessions run entirely outside your endpoints, so organizations that depend on endpoint visibility should account for this before rollout.[7]
