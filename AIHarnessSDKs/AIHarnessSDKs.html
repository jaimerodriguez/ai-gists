Building an AI Harness: OpenAI's Agent SDKs vs. Anthropic's Claude Agent SDK

An architect's comparison of the OpenAI Agents SDK, the Codex SDK, and the Claude Agent SDK as foundations for a first-class, self-verifying, memory-forming agent harness — including what each SDK withholds relative to its in-product sibling.

Research current as of July 9, 2026  ·  Sources: vendor docs, GitHub repos & issue trackers, HN/Reddit/practitioner blogs

If you are building your own agent harness — a runtime that runs an agent loop, gates its actions, verifies its work, persists memory, and learns across sessions — you face a choice that is not symmetric. Anthropic ships one SDK that is its product: the Claude Agent SDK is the Claude Code engine exposed as a library, with the harness (tools, permissions, hooks, compaction, memory, skills, subagents) already inside. OpenAI ships two SDKs that bracket the same territory from opposite ends: the Agents SDK is a deliberately thin orchestration framework that has been progressively growing harness organs (sandboxes, filesystem tools, memory, compaction — most of them beta as of April 2026), while the Codex SDK is the full Codex coding harness embedded as a subprocess, powerful but sealed — extensible out-of-process via hooks, MCP, and config, but with no in-process tool registration, no system-prompt override, and no official approval callback.

The compressed verdict, developed in Part III:

  • Most control per line of code for a custom harness: Claude Agent SDK. Its permission stack (modes → hooks → deny/ask/allow rules → canUseTool) and its hook system (which can block, rewrite inputs, replace outputs, inject context, and veto completion) have no equivalent on the OpenAI side. Its costs: Anthropic-model lock-in, a CLI-subprocess architecture that resists serverless deployment, an opaque bundled system prompt, and 0.x version churn with documented regressions.
  • Most architectural freedom: OpenAI Agents SDK. Code-first, model-agnostic (via LiteLLM/adapters), the cleanest multi-agent orchestration primitives (handoffs, agents-as-tools), and a serializable RunState that makes human-in-the-loop pauses survive process restarts. Its costs: you assemble the harness yourself — verification loops, permission policy, compaction, and memory are opt-in or DIY, and the harness-shaped parts (sandbox agents) are explicitly beta.
  • Closest analog to the Claude Agent SDK: the Codex SDK, not the Agents SDK. Both wrap their vendor's coding-agent harness as a subprocess. But the Codex SDK trades extensibility for kernel-level sandbox guarantees, and its cloud features (parallel tasks, best-of-N, hosted PR review) are unreachable from API-key auth entirely.
  • Portability from the in-product offering: Claude Code → Claude Agent SDK is a low-loss migration (mind two switches: the claude_code system-prompt preset and settingSources); Codex CLI → Codex SDK is nearly lossless locally because the SDK drives the same binary — but the Codex cloud product layer does not come along at all. On both sides, what you rebuild is the entire interactive UX.
  • For the self-verifying, memory-forming harness specifically: Anthropic currently offers more native leverage (Stop hooks that block completion, subagent evaluators, default-on auto-memory, skills-as-learning, automatic compaction) plus a published design doctrine; OpenAI reached first-party memory/compaction/skills primitives in 2026 but they live in the beta sandbox-agent stack, roughly one product cycle behind.

Part IThe OpenAI offering: two SDKs, two philosophies

OpenAI's agent story is fragmented by design. The Agents SDK (openai-agents, Python and TypeScript, open source) is a general-purpose orchestration framework descended from the experimental Swarm project. The Codex SDK (@openai/codex-sdk on npm; openai-codex on PyPI, beta) embeds the full Codex coding agent — the same Rust harness that powers Codex CLI — into your application. A third artifact, AgentKit (Agent Builder + ChatKit, launched October 2025), is already being dismantled: Agent Builder was deprecated on June 3, 2026 and shuts off November 30, 2026, with the code-first Agents SDK as the designated migration path. Understanding the division of labor between the two surviving SDKs is the prerequisite for comparing either to Anthropic's offering.

1. The Agents SDK: a thin framework growing a harness

The Agents SDK's founding pitch — "a lightweight, easy-to-use package with very few abstractions" — remains accurate for its core. An Agent is instructions + tools + handoffs + an optional typed output; a Runner executes the loop (call model → execute tools → repeat until a text-only final output); everything else is code-first Python/TypeScript control flow rather than a graph DSL. Launched March 11, 2025 alongside the Responses API, it sits at Python v0.18.0 / JS v0.13.0 as of July 2026 — still 0.x, with a stated policy that minor bumps may break public interfaces.

Core primitives

  • Handoffs transfer the whole conversation to another agent, exposed to the model as transfer_to_<agent> tools, with input filters to reshape the history the receiving agent sees. Agents-as-tools (agent.as_tool()) instead invoke a sub-agent as a scoped call without transferring the thread — the closest thing to Claude-style subagents, including max_turns caps, output extractors, and approval gates.
  • Guardrails are gate-shaped validators: input guardrails (first agent only) and output guardrails (last agent only) that raise a "tripwire" exception on violation. A separate, newer layer of tool guardrails can skip a tool call or replace its output — the closest the SDK gets to a policy engine. Guardrails validate and block; they never rewrite content.
  • Sessions are pluggable history stores: SQLite, Redis, SQLAlchemy, MongoDB, Dapr (30+ backends), an encrypted/TTL wrapper, server-side Conversations API sessions, and AdvancedSQLiteSession with conversation branching. The custom-backend interface is four methods. The JS SDK's built-in set is notably thinner (in-memory and Conversations API).
  • Structured outputs via Pydantic/Zod types on the agent's final output, with error handlers for malformed results.
  • Tracing is on by default, ships to a free OpenAI Traces dashboard, and has roughly thirty documented third-party processor integrations (LangSmith, Langfuse, Datadog, Braintrust, MLflow, …). It is the SDK's most consistently praised operational feature.
  • Model-agnosticism: any provider via LiteLLM (100+ models), custom ModelProviders, or the Vercel AI SDK adapter in JS — with the caveat that hosted tools, hosted MCP, and compaction require the OpenAI Responses API, so the "portable" subset is smaller than the full feature set. Practitioners report the bridge is leaky in practice (Responses-only features rejected on non-OpenAI backends).

Hooks, approvals, and the enforcement gap

This is the part most relevant to a harness builder, and the part most often misunderstood. The Agents SDK's lifecycle hooks (on_agent_start, on_tool_start/end, on_handoff, on_llm_start/end, …) are observation-only — they return nothing and cannot veto or modify an action. Enforcement is distributed across several narrower mechanisms instead:

  • needs_approval predicates on tools (function tools, shell, apply-patch, MCP servers, agents-as-tools) pause the run and surface interruptions; you call state.approve()/reject() and resume. Crucially, RunState serializes to a string, so an approval can span a process restart, a queue, or a human's lunch break. This is a genuinely strong human-in-the-loop primitive — arguably the best-designed one in either ecosystem for server-side deployments.
  • Tool guardrails can skip calls or replace outputs; tripwire guardrails can abort runs; call_model_input_filter can rewrite what the model sees per call.
  • There is no single interceptor that sees every action type uniformly — handoffs, for example, can be disabled but not vetoed by a guardrail — and there is no built-in permission-rule configuration format, no allowlist persistence, and no prompt UI. The plumbing is yours to assemble into a policy.

The April 2026 pivot: sandbox agents

On April 15, 2026, OpenAI announced "the next evolution of the Agents SDK": a beta SandboxAgent stack that pairs an agent with an isolated, persistent workspace. This is OpenAI retrofitting a Codex-shaped harness onto the thin framework, and it is widely read as convergence toward the Claude Agent SDK's feature set. It brings:

  • Filesystem and shell capabilities (ApplyPatchTool, ShellTool) with Docker and seven hosted sandbox providers (E2B, Modal, Daytona, Runloop, Blaxel, Cloudflare, Vercel);
  • Compaction as a default capability, plus OpenAIResponsesCompactionSession for ordinary sessions;
  • Memory, a capability that "distills lessons from prior runs into files inside the sandbox workspace" — generated as a pre-stop hook, steerable via MemoryGenerateConfig — OpenAI's first first-party learning-from-experience primitive;
  • Skills, lazy-loaded from .agents/skills/ directories using the same open SKILL.md standard Anthropic originated, plus AGENTS.md instructions and workspace snapshots for resumable long runs.
Caveat for architects

Everything harness-shaped in the Agents SDK is explicitly beta ("expect details of the API, defaults, and supported capabilities to change"). Outside the sandbox stack, ShellTool runs unsandboxed on your host, ApplyPatchTool requires you to implement the editor, and a plain Agent + SQLiteSession will overflow the context window with no automatic compaction. The stable core of this SDK is orchestration; the harness is the new, moving part.

What the Agents SDK does not provide

  • No permission prompt surface, rule format, or persisted user preferences — only approval plumbing.
  • No verification loop, plan mode, TODO tracking, or "block completion until checks pass" primitive.
  • No unified conversation+filesystem checkpoint/rewind (RunState covers approvals; sandbox snapshots cover workspaces; nothing joins them).
  • No long-term memory outside the beta sandbox Memory capability; no CLI/TUI; guardrails never rewrite.

2. The Codex SDK: the Codex harness as a library

The Codex SDK is a different animal: not a framework but an embedding of the complete Codex agent. The TypeScript SDK spawns the open-source Rust CLI and exchanges JSONL events over stdio; the beta Python SDK (openai-codex) speaks JSON-RPC to the Codex app-server — the same protocol the IDE extension uses. Either way, you get the harness OpenAI ships to millions of Codex users: the agent loop, tools, sandboxing, session persistence, compaction, skills, and subagents, "without extra tuning."

API surface

  • startThread() / resumeThread(id) (threads persist as JSONL rollouts under ~/.codex/sessions), thread.run() and runStreamed() yielding typed events (command_execution, file_change, mcp_tool_call, agent_message, reasoning, web_search, todo_list, …).
  • Structured output per run via JSON Schema or Zod; image inputs supported.
  • A config object flattened into --config overrides, meaning anything config.toml supports is reachable from the SDK — approval policies, sandbox modes, MCP servers, model providers, feature flags. This indirection is the SDK's main extension mechanism.

The harness you inherit

  • Kernel-level sandboxing — the headline differentiator. macOS Seatbelt, Linux/WSL2 bubblewrap, and native Windows Sandbox enforce filesystem and network isolation below the harness. As one comparison put it: "Codex enforces in the kernel; Claude Code enforces in the harness… if you want a hard guarantee that it cannot touch network or files outside its workspace, Codex's model is what you want." Modes: read-only, workspace-write (default), danger-full-access; network access is a separate switch with a newer domain-level allow/deny proxy.
  • Approval policies: untrusted / on-request / never, plus 2026's granular per-category routing (sandbox escalations, MCP elicitations, permission requests, skill approvals) and — notably — approvals_reviewer = "auto_review", which delegates escalation adjudication to an automatic reviewer agent instead of a human.
  • Lifecycle hooks (rolled out March–May 2026): SessionStart, UserPromptSubmit, PreToolUse, PermissionRequest, PostToolUse, Pre/PostCompact, SubagentStart/Stop, Stop. PreToolUse hooks can deny or rewrite tool inputs; PermissionRequest hooks can auto-allow/deny prompts; PostToolUse can replace output. These are configured in hooks.json/config.toml and executed as external commands — a file-and-subprocess contract, not an in-process callback. Third-party analysis reports only command-type handlers actually execute and PreToolUse interception is narrower than Claude's (single-source claim; medium confidence).
  • AGENTS.md layered instructions (global → repo root → cwd, 32 KiB combined cap), skills following the open SKILL.md standard (including "Record & Replay," which turns demonstrated workflows into reusable skills), subagents as TOML definitions (default 6 concurrent, depth 1), MCP client (stdio + streamable HTTP, per-tool approval modes) and MCP server (codex mcp-server exposes Codex itself as a tool — the documented bridge for orchestrating Codex from the Agents SDK), auto-compaction at ~85% of the context window, plan/todo tooling, and web search.
  • Model flexibility, surprisingly: the local harness accepts any provider speaking the Responses (and historically Chat Completions) wire API, with first-class --oss support for Ollama/LM Studio local models and Amazon Bedrock providers. The cloud/review/ChatGPT surfaces are OpenAI-models-only.

What the Codex SDK seals off

  • No in-process custom tools. There is no API to register a tool function on a thread; the sanctioned path is standing up an MCP server and wiring it via config. Compare: both the Claude Agent SDK and the Agents SDK let you define tools as plain functions in your process.
  • No supported system-prompt override. You steer with AGENTS.md, skills, and subagent instructions; the base prompt is the harness's.
  • No official approval callback in the SDK. The app-server protocol demonstrably carries interactive approval requests (community Elixir and Go SDKs implement approve/deny callbacks on it), but the official TS/Python SDKs expose policy-first configuration, not per-command interception. For a harness whose permission UX is the product, this is the single most consequential gap.
  • The entire Codex cloud layer — covered in Part III's portability section — is unreachable from API-key auth.

3. How OpenAI positions the two — and the AgentKit lesson

OpenAI's own guidance: use the Codex SDK for coding-focused threads; if Codex is one specialist inside a broader workflow, run codex mcp-server and orchestrate it from the Agents SDK. In other words: Agents SDK for the org chart, Codex SDK for the engineer. The two are converging — the Agents SDK grew Codex-style filesystem tools, sandboxes, skills, and memory in 2026 — but they remain distinct codebases with distinct maturity profiles: the Agents SDK's harness organs are beta; Codex's harness is battle-tested but its programmable surface is thin.

The AgentKit story deserves a paragraph in any lock-in analysis. Agent Builder — the visual workflow canvas launched at DevDay in October 2025 with enterprise fanfare — was deprecated eight months later (June 3, 2026; shutdown November 30, 2026). The community read was pragmatic rather than panicked, and the lesson practitioners drew is directly relevant here: code-first SDK layers survived; the hosted builder didn't. Anything you build on OpenAI's agent stack should assume the SDK is the durable surface and hosted conveniences are not.

4. What practitioners say

Agents SDK — praise

  • Simplicity is the dominant theme from launch to today: "one of the few agent abstractions I've seen that actually seems intuitive" (HN launch thread, March 2025); "handoff patterns and guardrails in under 100 lines of code" (2026 comparisons). Even roundups that recommend alternatives concede it is the fastest zero-to-agent path.
  • Built-in tracing and guardrails are repeatedly called "the minimum viable requirements for any production agent," with teams reporting materially faster debugging versus bolting observability on.
  • Real production adopters: Coinbase (prototyped its AgentKit wallet product on it in hours; later ~1,200 internal agents), Box (enterprise document workflows), Ramp, and a first-party Temporal integration as the de-facto durability path.

Agents SDK — criticism

  • Lock-in nuance: "OpenAI models only… a dealbreaker" is the most-repeated critique — technically unfair (LiteLLM exists) but directionally right, since the best features ride the Responses API and the multi-provider bridge has documented translation bugs.
  • Weak persistence/memory story pre-2026: "OpenAI agents forget everything between runs"; the GitHub issue tracker's dominant cluster is sessions/memory, including users explicitly requesting "a native compact feature similar to what Claude Code does."
  • "Optimized for demos, not production": missing resource management for long-running agents, failure isolation, and cost controls against runaway loops; guardrail/handoff behavior drifting between local tests and production.
  • 0.x churn: upgrades have broken Azure deployments and pinned-dependency chains; the versioning policy explicitly permits breaking changes in minor bumps. Maintainer responsiveness, however, is consistently rated good, with fast triage and weekly-to-biweekly releases.

Codex CLI/SDK — praise

  • Kernel sandboxing as a hard guarantee; generous flat-rate limits ("I've never hit my $20 plan limit"); token efficiency (~2–4× fewer tokens per task in community measurements); first-try reliability for well-scoped tasks; and local/OSS model support that made it a favorite in r/LocalLLaMA circles.
  • Rapid harness catch-up in 2026: hooks, subagents, skills (Simon Willison praised the progressive-disclosure design), and Windows-native support all landed within months.
  • Vercel's AI SDK 7 shipping a Codex harness adapter alongside Claude Code's is read as third-party validation that the Codex SDK is a credible programmable harness.

Codex CLI/SDK — criticism

  • Software quality churn is the defining complaint: severe recurring memory leaks (75 GB+ sessions; a 172 GB runaway), a persistent Windows sandbox + MCP instability cluster, apply-patch regressions in specific version bands, and a meta-issue asking for changelogs because features get deprecated while docs still reference them.
  • A security-flavored one worth flagging: MCP tools have been reported not to respect the sandbox (network and file writes even in read-only mode — openai/codex #7635). For a harness whose pitch is kernel enforcement, that is a material asterisk.
  • Approval prompts ignoring approval_policy="never" in some versions; lossy repeated compaction; session-resume gaps (largely fixed through 2026).
  • Skepticism toward OpenAI's own "harness engineering" marketing: the June 2026 post claiming an internal beta built with "0 lines of manually-written code" drew heavy HN pushback on metrics and verifiability.
Part I verdict — the OpenAI offering standalone

The Agents SDK and Codex SDK complaints are different in kind, and conflating them is the most common analytical error in this space. Agents SDK criticism targets framework design (memory, persistence, lock-in shape); Codex criticism targets software quality and limits (leaks, sandbox bugs, release hygiene) while the harness concept itself is mostly praised. For a custom harness: the Agents SDK gives you clean orchestration bones and asks you to grow the organs; the Codex SDK gives you a complete organism you may only steer from outside. The documented composition pattern — Agents SDK orchestrating Codex via MCP — is OpenAI's implicit admission that neither alone covers the territory the Claude Agent SDK covers in one package.

Part IIThe Anthropic offering: the Claude Agent SDK

Anthropic's line is one product deep and one philosophy wide: the Claude Agent SDK (renamed from "Claude Code SDK" in September 2025) is Claude Code, exposed as a TypeScript and Python library. There is no thin-framework counterpart; if you want orchestration bones without the harness, you drop to the raw Messages API. The SDK's pitch — "same capabilities, different interface" — is close to literal: the agent loop, tool suite, permission engine, hook system, compaction, memory, skills, and subagents that ship in the product ship in the library. The design doctrine behind it is published and unusually explicit: an agent loop of gather context → take action → verify work → repeat, with the filesystem and bash as the agent's primary interface to the world.

1. Architecture: Claude Code as a subprocess

When your code calls query(), the SDK spawns a Claude Code process and talks to it over stdio; one agent session maps to one subprocess that owns the shell, the working directory, and JSONL session transcripts on local disk. Two important updates to the folk wisdom about this architecture:

  • The Node.js dependency is gone. Both SDKs now bundle a native Claude Code binary pinned to the SDK package version — Python 3.10+ is the Python SDK's only runtime requirement. The widely-cited "it spawns node and breaks in Electron/slim containers" pitfalls describe the 2024–2025 architecture.
  • The subprocess is swappable in principle. Python exposes cli_path and an abstract Transport class, so the stdio child can be replaced with, e.g., a remote connection.

The architecture's real costs remain: every agent is a long-lived process (~1 GiB RAM / 5 GiB disk / 1 CPU guidance) tied to local state, which is why serverless and microVM deployments have been the loudest pain point. Anthropic's mitigations are recent: SessionStore adapters mirror transcripts to S3/Redis/Postgres for cross-host resume, and the hosted "Claude Managed Agents" product (public beta April 2026) exists precisely because self-hosting this was hard.

The built-in tool suite is the product's: Read/Write/Edit, Bash (plus Monitor for watching background processes), Glob/Grep, WebSearch/WebFetch, the Agent tool for subagents, Skill, task-tracking tools, AskUserQuestion, on-demand tool loading via ToolSearch, and (TypeScript) a Workflow tool for script-driven orchestration of large agent fleets. Structured outputs validate against JSON Schema with automatic re-prompting on mismatch. Read-only tools run concurrently; mutating tools serialize.

2. The permission stack and hooks

This is the SDK's crown jewel and the single largest capability gap versus everything OpenAI ships. It is a layered pipeline, evaluated in a documented order for every tool call:

  1. PreToolUse hooks — in-process callbacks (or settings-file commands) that can deny outright, allow, escalate to "ask," rewrite the tool's input, or defer;
  2. Deny rules — declarative patterns (Bash(rm *)) that block even in bypassPermissions mode;
  3. Ask rules — patterns that force escalation to the callback;
  4. Permission mode — six modes: default, dontAsk, acceptEdits, plan (read-only exploration; writes never auto-approve), bypassPermissions, and auto, in which a model classifier adjudicates each call. Modes can change mid-session;
  5. Allow rules — declarative auto-approval patterns;
  6. canUseTool — your programmatic final arbiter, which can approve, deny with a message, or approve with modified input.

The documented footgun: auto-approved tools never reach canUseTool, so checks that must run on every call belong in a PreToolUse hook (recent SDK versions emit a warning when your callback is shadowed). The design consequence for a harness builder: declarative policy, interactive escalation, and programmatic override are all first-class and composable — you implement a permission UX, not a permission engine.

The hook system extends far beyond permissions. Events include PreToolUse, PostToolUse (which can replace tool output and inject context back to the model), PostToolUseFailure, UserPromptSubmit, Stop, SubagentStart/Stop, PreCompact, PermissionRequest, Notification, and (TypeScript only) session-lifecycle, config-change, task-completion, and worktree events. Three properties matter most for harness design:

  • Hooks can block completion. A Stop hook returning a block decision forces the agent to keep working, with the reason fed back to the model — the native "not done until verification passes" primitive.
  • Hooks can mutate. Input rewriting, output replacement, and context injection turn hooks into a feedback channel, not just a veto.
  • Hooks are in-process functions (async supported, parallel execution, most-restrictive-wins), so your policy code shares state with your application. Python lags TypeScript on several event types — a real asymmetry to check before choosing a language.

3. Harness organs: context, memory, subagents, skills, sandbox

Context management

Compaction is automatic and default-on: when context nears the limit, the harness summarizes older history (emitting a compact_boundary marker), re-injects startup content, and continues. Prompt caching is automatic and aggressive, with a documented option to restructure the system prompt so identical agent fleets share cache entries. Sessions persist as JSONL with resume, forking, and (TypeScript) opt-out. File checkpointing is SDK-exposed: rewindFiles(uuid) restores files to a message-boundary checkpoint — tracking edits made via file tools, though not via bash, and files only, not conversation state. A 1M-token context window is available on current models.

Memory

Three distinct mechanisms, often conflated:

  • CLAUDE.md — human-authored standing instructions, loaded hierarchically (user → project → local, plus path-scoped rule files), when settingSources includes the matching source.
  • Auto-memory — agent-authored, on by default, SDK-exposed: a per-project memory directory with a MEMORY.md index loaded into every session, which the agent maintains with its ordinary file tools. This is the closest thing either vendor ships to default-on cross-session learning. Subagents can carry their own memory scopes.
  • The API-level "memory tool" (a Messages API beta where your application implements the storage backend) — notably not part of the Agent SDK surface. First-party managed cross-session memory landed in the hosted Managed Agents product instead, a fact worth registering in any build-vs-buy analysis.

Subagents, skills, plugins

  • Subagents are definable programmatically (name, system prompt, tool allowlist/denylist, per-agent model, per-agent MCP servers, memory scope, permission mode, effort) or via filesystem markdown. They run concurrently and in the background by default, isolate context fully (fresh window; only the final message returns), can nest to depth five, and are resumable by ID. This is materially richer than the Agents SDK's agents-as-tools and has no Codex SDK equivalent (Codex subagents exist but are TOML-configured six-thread affairs).
  • Skills are filesystem-only (SKILL.md with progressive disclosure) — no programmatic registration API — and are the load-bearing mechanism in Anthropic's "agents codify their own patterns of behavior into reusable capabilities" learning story.
  • Plugins bundle skills/agents/hooks/MCP config, but the SDK only loads local paths — marketplace installation is product-side.

Sandboxing

OS-level sandboxing exists (macOS Seatbelt; Linux/WSL2 bubblewrap with a proxy-based network layer supporting per-domain allow/deny and credential masking) and is configurable from SDK options. Two honest caveats versus Codex: there is no native Windows sandbox (WSL2 only), and reads are permissive by default (whole disk minus denials — credentials readable unless configured). The default network proxy does not terminate TLS, with a documented domain-fronting exfiltration caveat. Codex's kernel-first posture is stronger by default; Claude's is more configurable and more deeply integrated with the permission system.

MCP and custom tools

MCP support covers stdio, SSE, and streamable HTTP as a client, plus in-process SDK MCP servers: custom tools defined as plain functions in your application, no subprocess, with annotations that opt them into parallel execution. MCP tool schemas are deferred out of context by default and loaded on demand — relevant at fleet scale. OAuth flows are your responsibility, and claude.ai-account connectors auto-load unless explicitly suppressed (a multi-tenant gotcha).

4. What practitioners say

Praise

  • "Claude Code as a library" is the dominant theme: "The built-in tools, permissions, and session handling collapse weeks of harness work… nothing else matches it for coding-agent ergonomics" (June 2026). Hands-on reports cite parallel subagents delivering 6.7× faster document processing with 40% less context, and MCP integrations wired "in an afternoon."
  • The harness demonstrably matters: at Anthropic's AI Engineer Code Summit, Opus 4.5 scored 78% on the CORE benchmark inside the Claude Code harness versus 42% in alternative frameworks — the number most often quoted in the SDK's favor.
  • Ecosystem gravity: Apple's Xcode integrated the SDK (February 2026); Microsoft's Agent Framework ships an official Claude package; Notion, Rakuten, and Sentry run on the managed offering; Anthropic's own Cowork product is built on it. Search demand grew ~300× from May 2025 to April 2026.

Criticism

  • The subprocess architecture draws the sharpest technical fire: "just a wrapper around the CLI," awkward in Electron, and hostile to serverless — one HN practitioner self-hosting in microVMs described crashes in streaming mode and "quite fragile" custom lifecycle plumbing. (Partially mitigated since: bundled native binary, SessionStore adapters.)
  • Opacity: "the loop is a black box you steer rather than code you step through." A third-party project exists solely to decompile and track the bundled system prompt (110+ fragments across 126 versions). Config precedence surprises (a stray ANTHROPIC_API_KEY in user settings silently outranking your env var) are documented pitfalls.
  • Version churn with teeth: the TS changelog shows repeated breaking minor releases; the worst community report is a 3× token-cost regression between two 0.1.x versions on identical tasks. The chronic GitHub clusters are subscription-vs-API auth and local-filesystem state vs. cloud deployment.
  • The May–June 2026 billing whiplash is the biggest trust event: Anthropic announced SDK/headless usage would move off subscription limits to a separate credit pool, drew furious HN reaction ("yet another developer-hostile decision"), then paused the change on June 15 — the day it was to take effect. Post-pause sentiment holds that metered SDK billing "will eventually happen." The standing policy that third-party products may not offer claude.ai login to their users is unambiguous; individual use of one's own subscription with the SDK works mechanically and currently draws from a plan-included Agent SDK credit.
  • Model lock-in is architectural: officially Anthropic models only (Anthropic API, Bedrock, Vertex, Foundry). Proxy workarounds exist and are unsupported; comparison posts route multi-model teams to LangGraph or the Agents SDK.
Part II verdict — the Anthropic offering standalone

The Claude Agent SDK is the most complete agent harness available as a library, and the only one whose permission and hook systems were designed for a harness builder rather than retrofitted. Its trade is equally clear: you inherit a vendor's opinionated, partially opaque runtime — its models, its subprocess architecture, its release cadence, and its billing politics. Teams that love it treat it as an appliance they steer; teams that fight it are usually trying to make it something it is not (a stateless library, a multi-model router, or a fully transparent loop).

Part IIICross-comparison

The fair comparison is triangular. The Claude Agent SDK's true peer is the Codex SDK — both are their vendor's coding harness wrapped as a subprocess-driving library. The Agents SDK is a different species (an orchestration framework) that has been growing harness organs; it competes with the Claude Agent SDK only in the sense that both are places you could choose to build. Where a single winner matters below, the Codex SDK is treated as the primary OpenAI comparator, with the Agents SDK scored alongside.

2. Permissions and hooks, head to head

The user-facing question for a harness builder is: when my agent is about to do something, what can my code know, veto, or change — and how much of that machinery do I have to build?

Claude Agent SDK: a permission engine you configure. The six-step pipeline (hooks → deny rules → ask rules → mode → allow rules → callback) means declarative policy, pattern-matching, interactive escalation, and programmatic override compose without custom glue. The hook system doubles as a verification and feedback channel: PostToolUse can rewrite what the model sees; Stop can refuse to let the agent finish. Everything runs in your process with your state. The gaps are ergonomic rather than structural: the auto-approval/callback shadowing footgun, Python's missing hook events, and the fact that a "permission prompt" is still a UX you must render yourself.

Codex SDK: a permission policy you select. Approval policies, granular category routing, execpolicy command rules, and sandbox modes give strong declarative control — arguably stronger defaults than Claude's, since the kernel sandbox backstops everything. The 2026 hooks close much of the interception gap (deny/rewrite in PreToolUse, auto-decide in PermissionRequest). But the contract is files and subprocesses, not in-process callbacks; the official SDKs expose no per-command approval API even though the underlying app-server protocol carries one (community SDKs in Elixir and Go prove it). If your harness's permission UX is central — an approval inbox, a policy learned from user decisions, an audit trail with context — you will either work at the protocol layer or accept Codex's own adjudication surfaces (including its novel auto-reviewer agent).

Agents SDK: permission primitives you assemble. Ironically, it has the single best-engineered approval primitive of the three — needs_approval interruptions with a serializable RunState that survives process restarts, ideal for server-side approval queues — but nothing around it: no rule format, no modes, no persisted preferences, no uniform interceptor, and lifecycle hooks that cannot enforce anything. You get a good axle and build the rest of the cart.

Verdict — permissions & hooks

For a harness whose identity includes a custom permission model: Claude Agent SDK, clearly. For hard containment guarantees with minimal ceremony: Codex. For asynchronous, restart-surviving human approval flows in a distributed system: the Agents SDK's RunState is the best single primitive, at the cost of assembling everything else.

Codex (product) → Agents SDK

For completeness: this migration is a rebuild, not a port. The Agents SDK shares no config surface with Codex; you would recreate the harness from beta parts (sandbox agents, shell/apply-patch tools, skills, memory) or bridge to the real one via codex mcp-server. The bridge pattern is officially documented and is the honest answer when you want Agents SDK orchestration and Codex's harness.

Verdict — portability & completeness vs. the in-product offering

Anthropic exposes more of its product through the SDK than OpenAI does — the notable Claude Code exclusives are collaboration/UX features, while Codex's exclusives include genuine capability (cloud parallelism, best-of-N, hosted review). Symmetrically, both SDKs withhold the interactive UX layer, and both vendors reserve their richest managed offerings (Claude Managed Agents; Codex cloud) for their own perimeter. A personal harness loses least by building on the Claude Agent SDK; a harness that wanted Codex's cloud muscle keeps a dependency on the Codex product no matter what.

4. The first-class harness lens: verify, remember, learn

Assume the target is a harness that (a) verifies its own work, (b) saves memories, and (c) learns from them. How much does each SDK give you natively, and what does the field's experience say about the remainder?

Verification

Anthropic has both the primitives and a published doctrine. The primitives: Stop hooks that block completion until checks pass; PostToolUse hooks that inject feedback (lint output, test failures) directly into the loop; isolated subagent evaluators for adversarial review; and Monitor/Bash for driving real end-to-end checks. The doctrine, from Anthropic's engineering posts: prefer deterministic, rules-based feedback over LLM judges; give the agent browser automation because code-only inspection misses real failures; and — critically — separate the evaluator from the generator, because agents asked to grade their own work "respond by confidently praising the work" even when it is mediocre. Documented failure modes (premature victory declarations, features marked done without verification) match what practitioners report, and a real footgun exists: a malformed Stop hook once looped an agent for a full session. Independent practitioners corroborate the doctrine's LLM-judge skepticism ("slow, expensive, unreliable" versus deterministic checks).

On the OpenAI side, verification splits by SDK. The Agents SDK offers gate-shaped guardrails (pass/fail tripwires, optionally LLM-judge-powered) and clean code-first loops you write yourself — a generator→judge→retry loop is a documented pattern, not a primitive. Codex offers verification as a product — the hosted PR review and local /review presets are excellent for the code-review workflow specifically — plus hooks that can replace tool output with feedback; whether its Stop hook can block completion Claude-style is unconfirmed. Telemetry-grounded self-verification (giving the agent a query DSL over runtime traces so it "verifies against runtime evidence rather than guessing") is an emerging pattern on both stacks.

Memory

Both vendors converged on the same architectural bet — file-shaped memory the agent reads and writes itself — which the evidence favors over vector-store memory layers. Anthropic ships it default-on (auto-memory with a MEMORY.md index) with CLAUDE.md for human-authored standing rules; OpenAI's equivalent (the sandbox Memory capability distilling "workflow lessons" as a pre-stop hook; Codex's off-by-default memories flag) arrived in 2026 and is one maturity notch behind. Neither SDK ships retrieval-backed semantic memory, and the third-party memory-layer market (Mem0, Zep, Letta, LangMem — all of which integrate with both SDKs) should be evaluated skeptically: the LOCOMO benchmark most vendor claims rest on was shown to accept 62.8% of intentionally wrong answers, vendors publicly dispute each other's evaluations, and academic work finds no advanced memory system consistently beating a plain RAG baseline. The credible pattern is boring: explicit, inspectable files under git, with a periodic consolidation pass — which both harnesses can do, and Claude's does by default.

Learning

The proven learning loop in the field is skills-as-accumulated-capability: after each run, distill what worked into a SKILL.md (or lessons file) that future runs load on demand. Anthropic states this as an explicit goal ("agents create, edit, and evaluate Skills on their own, codifying their own patterns of behavior"), has a marquee production example (Warp's agents rewriting their own skills from human corrections), and a mature community methodology (compound engineering: plan → work → review → compound, with lesson files grepped before acting). OpenAI adopted the same open SKILL.md standard in Codex (including Record & Replay skill generation from demonstrations) and added skills to the Agents SDK's sandbox stack — the mechanics are converging fast, but the doctrine and worked examples are, as of mid-2026, roughly "one cookbook in." For a learning harness, the standard's portability is the strategic fact: skills you author are the most vendor-portable asset in either ecosystem.

Context engineering

Claude's SDK does the most for you by default (automatic compaction, aggressive caching, subagent isolation, deferred tool schemas); Codex's harness does comparable work inside the CLI (with community-documented quality complaints at compaction boundaries); the Agents SDK does the least by default but now has the pieces. One dissenting note from Anthropic's own harness-design writing applies everywhere: for very long-running work, context resets with structured file handoffs beat in-place compaction — a clean-slate session reading a good progress file outperforms a summarized one, and counters "context anxiety" (models rushing as the window fills). A first-class harness should treat compaction as a safety net, not the architecture.

Verdict — the self-verifying, learning harness

Build-cost asymmetry, not capability ceiling, is the real difference. Everything described above is achievable on all three SDKs; on the Claude Agent SDK the verification loop, memory, and learning substrate are native or default-on, on Codex they are configuration plus product features, and on the Agents SDK they are your code. The deeper your ambitions for permission UX and verification enforcement, the more the Claude SDK's in-process hooks pay; the more you value containment guarantees and multi-model freedom, the more the OpenAI pairing pays.

5. Maturity, lock-in, cost, and operations

  • All three are 0.x. None has shipped a 1.0; all reserve the right to break you in minor releases, and all have done so (Agents SDK: dependency and default-model breaks; Codex: apply-patch and hook-config regressions, changelog hygiene complaints; Claude: removed session APIs, a reported 3× cost regression). Pin versions, read changelogs, and budget for churn regardless of choice.
  • Lock-in has different shapes. Claude's is model lock-in (architectural, by policy). The Agents SDK's is platform-gravity lock-in (portable in principle; best features Responses-only). Codex's is perimeter lock-in (open local harness, gated cloud). The AgentKit deprecation and the Claude billing whiplash are the two cautionary tales: hosted conveniences and pricing terms are the least durable parts of both ecosystems — the code-first SDK surfaces have been the stable ground.
  • Billing for an always-on harness: OpenAI steers programmatic use to API keys (Codex cloud excepted — ChatGPT auth only); community sentiment credits Codex plans with the more generous flat-rate limits for product use. Anthropic currently includes an Agent SDK credit with paid plans for individual use, after pausing a controversial metering change; multi-user distribution on subscription auth is prohibited on both sides. Token economics in long loops are dominated by cache discipline on both stacks (cached input ≈ 10% of fresh); compaction and session-resume behaviors that invalidate caches are documented cost traps on both.
  • Operational posture: the Claude SDK wants a persistent machine (long-lived subprocess, local state; SessionStore for cloud); Codex likewise runs local-first with the cloud as a separate product; the Agents SDK is the most natural fit for conventional server-side deployment (stateless-ish library, serializable state, Temporal integration for durability). If your harness must run in serverless or fleet-scale infrastructure, this row may outweigh several feature rows.
  • Ecosystem hedge: meta-harness layers now exist — Vercel AI SDK 7's HarnessAgent wraps Claude Code, Codex, and others behind one API, and startups abstract both SDKs as swappable backends. Designing your harness's internal interfaces so the vendor SDK is a replaceable driver is no longer speculative; it is the emerging industry pattern.

6. Recommendations

For the specific goal stated at the outset — a personal, first-class harness that verifies its work, saves memories, and learns from them:

  1. Default choice: build on the Claude Agent SDK — it is the only offering where the permission stack, blocking hooks, subagent evaluators, auto-memory, skills, and compaction are native, and the Claude Code → SDK migration preserves your accumulated assets (skills, hooks, MCP, CLAUDE.md, memory) nearly intact. Accept the trade knowingly: Anthropic models only, a subprocess runtime, and exposure to Anthropic's pricing politics.
  2. If containment or multi-model freedom is non-negotiable, pair the OpenAI SDKs: Codex SDK (or codex mcp-server) as the sandboxed executor, Agents SDK as the orchestration and approval layer. This buys kernel-level isolation and provider flexibility at the cost of assembling verification and permission UX yourself — and forgoing Codex cloud unless you keep ChatGPT-authenticated product surfaces in the loop.
  3. Whichever substrate you choose, keep the vendor-portable core vendor-neutral: skills (SKILL.md is now a cross-vendor standard), AGENTS.md/CLAUDE.md instructions, lesson/memory files under git, MCP servers, and deterministic verification scripts all port. The SDK-specific investment — hook wiring, permission policy, session plumbing — should live behind an interface you own, following the HarnessAgent-style pattern. The teams that regretted their choice were locked in by plumbing, not by assets.
  4. Design the harness around the field's three hard-won lessons: separate the evaluator from the generator (self-grading agents flatter themselves); prefer deterministic checks and real end-to-end exercise over LLM judges; and prefer explicit file-based memory with periodic consolidation over semantic memory layers whose benchmarks don't survive scrutiny.
